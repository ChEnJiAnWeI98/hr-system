## 第一章 使用说明

### 1.1 什么时候读这个文档

**只有当你要做以下任一类"数据变动"时，才需要查阅本文档**：

| 动作类别 | 具体场景 |
|---|---|
| **员工生命周期** | 新增员工 / 员工转正 / 员工异动 / 员工离职 / 批量导入 |
| **合同变动** | 签订合同 / 续签 / 终止 / 合同到期自动处理 |
| **组织变动** | 新增组织 / 撤销组织 / 员工迁移 / 编制调整 |
| **岗位变动** | 新增岗位 / 废弃岗位 / 岗位归属调整 |
| **字典变动** | 新增字典值 / 修改字典值 / 删除字典值 |
| **跨模块业务数据生成** | 生成新周期绩效 / 工资 / 考勤 / 培训 / 福利 / 保险数据 |
| **关联关系设置** | manager_id 重新指派 / 继任者名单 / 审批流节点 / 评估人名单 |

**不在上述场景内的代码改动**（例如改 UI、改按钮样式、加页面），**无需读本文档**。

### 1.2 使用流程

```
Step 1：识别你的动作 → 归入上表某一类
Step 2：查第二章"触发索引" → 找到对应的规则号清单（如 [A1, B3, F1]）
Step 3：按规则号去第三章查详细定义 → 对照检查
Step 4：发现异常 → 查第四章处置建议 → 修复
```

### 1.3 关键定义

#### 在职 vs 已离职
- **在职状态**：`regular`（正式）/ `probation`（试用）/ `transferring`（异动中）/ `seconded`（借调）/ `pending_onboard`（待入职）
- **离职流程中**：`offboarding` — 已发起离职申请，但**离职生效日未到**，仍是在职员工
- **已彻底离职**：`terminated` — 离职生效日已过，已走出公司

#### 历史数据 vs 当前数据（按日期判定）
- **历史数据**：数据所属日期 ≤ 员工离职日 → **100% 保留，永不删除**
- **当前/未来数据**：数据所属日期 > 员工离职日 → **不应存在**

举例：员工 4 在 2026-01-15 离职
- 2025-12 工资条 ✅ 保留（历史）
- 2026-01 工资条（1 日-15 日）✅ 保留（历史，含离职当月结算）
- 2026-02 工资条 ❌ 不应存在
- 2026 Q2 绩效档案 ❌ 不应存在

## 第三章 规则详细定义

### A. 员工自身状态一致性

#### A1 员工 status 与岗位 status 一致
- **规则**：员工所占岗位的状态必须与员工状态匹配
- **异常示例**：`terminated` 员工还占着"在编"岗位（岗位的 `occupiedBy` 还是这个员工 ID）
- **修复**：员工 terminated 时，岗位的 `occupiedBy` 清空或改为历史记录

#### A2 员工 status 与组织 status 一致
- **规则**：员工所在组织必须是 active 状态
- **异常示例**：员工所在部门已被撤销（`status=archived`），但员工还是 regular
- **修复**：组织撤销前必须强制迁移所有员工（参见 F4）

#### A3 员工 status 与合同 status 一致
- **规则**：员工在职（regular/probation）→ 必须有一份"履约中"的合同；员工 terminated → 合同应为"已终止"
- **异常示例**：员工 regular 但无生效合同；员工 terminated 但合同还是"履约中"
- **修复**：员工状态变更时同步合同状态（参见 F2）

---

### B. 时间一致性 ⭐（易发生隐性异常）

#### B1 合同截止日 ≥ 今天（状态为履约中时）
- **规则**：合同 status=履约中 时，endDate 必须 ≥ 今天
- **异常示例**：**合同截止 2026-01-05，但 status 还是"履约中"，员工还 regular**（用户的经典例子）
- **修复**：
  - 若员工实际已离职 → 合同改"已终止"
  - 若合同实际已续签 → 创建新合同，旧合同改"已终止（已续签）"
- **Mock 层实现（V1.1 起）**：员工池在 `src/mock/core/employeePool.ts` 的 `generatePoolContracts()` 中按以下规则生成合同：
  - 首签期限：1/2/3/5 年加权随机（3 年权重 55 占主流）
  - 首签到期 → 自动续签（续签期限 3/5 年），续签到期再续签，直到最新一份 endDate ≥ 今天
  - terminated 员工：合同对齐离职日，状态 = 已终止

#### B2 合同生效日 ≤ 今天（状态为履约中时）
- **规则**：合同 status=履约中 时，startDate 必须 ≤ 今天
- **异常示例**：合同 2026-06-01 生效，但 4 月 status 已是"履约中"
- **修复**：未生效的合同 status 应为"待生效"

#### B3 入职日 ≤ 今天（status 为在职类时）
- **规则**：员工 status ∈ {probation, regular, transferring, seconded} → hireDate 必须 ≤ 今天
- **异常示例**：入职日 2026-05-01，但 4 月 status 已是 regular
- **修复**：未入职员工 status 应为 `pending_onboard`

#### B4 离职日 ≤ 今天 ⇔ status = terminated
- **规则**：terminationDate ≤ 今天 ⇔ status = terminated
- **异常示例**：
  - 离职日 2025-12-31，但 status 还是 regular
  - 离职日 2026-12-31（未来），但 status 已是 terminated
- **修复**：严格按日期自动推状态

#### B5 试用期未满 ⇔ status = probation
- **规则**：入职日 + 试用期月数 > 今天 ⇔ status = probation
- **异常示例**：入职 2025-08，试用期 6 个月（至 2026-02），但 2026-04 仍是 probation
- **修复**：自动转正 or 手动延长试用期
- **Mock 层实现（V1.1 起）**：员工池 post-processing 函数 `applyProbationAutoRegularization()` 会扫描 probation 员工，入职 > 6 个月者自动转为 regular 并补 regularizationDate = 入职日 + 6 个月

#### B6 绩效档案日期 ≤ 员工离职日
- **规则**：若员工已 terminated，绩效档案的 `cycleStartDate` 必须 ≤ terminationDate
- **异常示例**：员工 2025-12 离职，却有 2026 Q2 绩效档案
- **修复**：生成新周期绩效时过滤已离职员工

#### B7 工资条月份 ≤ 员工离职月
- **规则**：若员工已 terminated，工资条 `payrollMonth` 必须 ≤ terminationDate 所在月
- **异常示例**：员工 2026-01-15 离职，却有 2026-02 工资条
- **修复**：生成新工资批次时过滤已离职员工
- **边界**：离职当月的工资条**必须保留**（含最后结算）

#### B8 考勤/加班/请假/外勤/补卡日期 ≤ 员工离职日
- **规则**：若员工已 terminated，这些记录的日期必须 ≤ terminationDate
- **异常示例**：员工 2026-01 离职，却有 2026-03 的打卡记录
- **修复**：生成新考勤时过滤已离职员工

### C. 跨模块关联一致性

#### C1 manager_id 指向的员工必须在职
- **规则**：员工的 managerId 指向的员工必须 status ∈ 在职类
- **异常示例**：李四的主管王五已 terminated
- **修复**：主管离职时触发下属主管重新指派（参见 F1）

#### C2 继任者名单中无已离职员工
- **规则**：岗位的继任者清单 successorIds 中不能有 terminated 员工
- **异常示例**：CTO 岗位的继任者是已离职的员工 4
- **修复**：员工 terminated 时，扫描所有继任者清单剔除

#### C3 培训计划学员中无已离职员工
- **规则**：培训计划 participantIds 中不能有 terminated 员工（且计划开始日 > 离职日）
- **异常示例**：2026-04 培训计划学员名单包含 2026-01 离职员工
- **修复**：生成新培训计划时过滤 + 员工离职时扫描清除未开始计划

#### C4 福利/保险池中无已离职员工
- **规则**：福利发放对象、保险参保人列表中不能有 terminated 员工（针对离职后的新发放）
- **异常示例**：离职员工还在 2026-04 生日福利名单
- **修复**：生成新福利发放时过滤 + 员工离职时自动停保

#### C5 PIP / 360 / 1on1 / 校准 对象和发起人必须在职
- **规则**：这些业务的 subjectId, initiatorId, reviewerIds 必须都在职
- **异常示例**：已离职员工在某人的 360 评估人列表
- **修复**：生成新周期时过滤；进行中的流程中有人离职 → 替换评估人

#### C6 合同签署人必须存在
- **规则**：合同的 employeeId 必须在员工池中存在
- **异常示例**：合同显示签署人员工 ID 999，但员工池只到 200
- **修复**：导入合同时校验 employeeId

#### C7 审批流当前节点审批人必须在职
- **规则**：流程中的 currentApproverIds 必须在职
- **异常示例**：审批卡在已离职员工那里，流程永远走不下去
- **修复**：员工离职时自动转移待办 or 触发替代审批人机制

### D. 组织 / 编制一致性

#### D1 非叶子组织的 budgetCount = ∑ 子组织 budgetCount
- **规则**：父组织编制 = 所有子组织编制之和（rollup 语义）
- **异常示例**：父组织编制 100，子组织合计 120
- **修复**：父节点 budgetCount 改为自动聚合（不可直接编辑）

#### D2 组织 headCount ≤ budgetCount（除非允许超编）
- **规则**：实际在职人数 ≤ 编制数；超编需有显式标记 allowOverstaff=true
- **异常示例**：组织编制 10，实际在职 15，且无超编许可
- **修复**：触发超编告警 or 补齐编制 or 减员

#### D3 组织 headCount = 员工池中该组织的在职员工数
- **规则**：组织的 headCount 字段必须与员工池实时计算结果一致
- **异常示例**：组织 headCount=8，但员工池里该组织在职员工数=7
- **修复**：headCount 改为运行时动态计算（不存字段），或保存时同步

### E. 字典一致性 ⭐（易漏）

#### E1 员工引用的字典值必须在字典表中
- **规则**：员工档案里每个字典字段（education/marriage/politicalStatus 等）的 value 必须在对应字典中存在
- **异常示例**：员工 education 字段是"高中"，但字典里只有"初中/大专/本科/硕士/博士"
- **修复**：新增员工/修改字段时用 select 组件从字典取值

#### E2 字典删除/改名时引用方必须更新
- **规则**：修改字典项前必须扫描使用方
- **异常示例**：字典把"未婚"改为"单身"，但员工档案里还是"未婚"
- **修复**：修改字典前 grep 所有引用 → 一起改

#### E3 合同/异动/离职类型必须对应字典
- **规则**：合同的 contractType、异动的 transferType、离职的 offboardingType 必须是当前字典的合法值
- **异常示例**：合同类型写"实习"，但字典里只有"劳动合同/劳务合同/兼职合同"
- **修复**：输入时做校验

#### E4 新增字典值必须考虑历史数据初始化
- **规则**：新增字典项时，明确老数据如何处理（回填、保持 null、迁移默认值）
- **异常示例**：新加"孕期保护"状态，老员工该字段都是 undefined
- **修复**：新增时给出迁移策略

### F. 数据流转闭环 ⭐（核心）

#### F1 员工 regular → offboarding 的涟漪
发起离职申请时，必须同步：
- 合同 status → "待终止"
- **停发**离职生效日之后的新工资条（当月结算正常）
- **退出**离职日之后的新周期绩效考核
- **退出**离职日之后的新培训计划
- **退出**离职日之后的新福利发放
- manager_id 指向他的下属 → 重新指派主管
- 他作为审批人的流程 → 转移到替代审批人
- 他在的 PIP / 360 / 1on1 → 标记中止或转移

#### F2 员工 offboarding → terminated 的涟漪
离职办完时，必须同步：
- 合同 status → "已终止"
- 岗位 occupiedBy 清空（编制释放）
- 停社保公积金
- 最后一期工资结算
- 禁用系统账号（账号状态 → disabled）
- 历史数据全部保留

#### F3 合同到期的涟漪
合同到期时系统应推：
- 若到期前已签续签合同 → 原合同 status → "已续签"，新合同生效
- 若到期前未续签 → 触发到期告警 → HR 确认 → 合同 status → "已终止"，员工 status 联动 terminated

#### F4 组织撤销的涟漪
组织撤销前必须：
- 该组织所有员工必须先迁移（否则禁止撤销）
- 该组织下的岗位必须先归属其他组织或废弃
- 该组织的编制 budgetCount 归零
- 历史数据（曾经属于该组织的员工记录）保留

#### F5 岗位废弃的涟漪
岗位废弃前必须：
- 该岗位员工必须先调岗（否则禁止废弃）
- 编制 budgetCount 归零
- 继任者清单清空

## 第四章 违规处置建议

### 4.1 发现异常后的通用流程
1. **判断异常性质**：是"历史遗留无害"还是"当前业务有害"？
2. **历史遗留无害**（如 2020 年离职员工有 2020 年工资条）→ **不修**
3. **当前业务有害**（如 2026 新工资条有已离职员工）→ **必修**
4. **影响评估**：改动涉及哪些下游？（对照 F1~F5）
5. **修复+验证**：改完重跑检查脚本，确认异常已清 0

### 4.2 批量修复工具
推荐使用统一的**检查脚本**（见 `scripts/check-data-health.js` — 第 A 期建）
- 输入：整个 mock 数据
- 输出：按规则号分组的异常清单
- 用法：每次改完 mock 跑一遍，异常数应 = 0

### 4.3 修复原则速记
- **历史保留**：永远不删历史
- **按日期过滤**：判断"当前 vs 历史"用数据所属日期 vs 员工离职日
- **字典先行**：字典改动前必 grep 引用方
- **流转闭环**：状态变更必走 F 章的涟漪清单
