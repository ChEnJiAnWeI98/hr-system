## 一、设计理念

### 1.1 核心定位

> **AI 是效率工具，不是判断工具。**

所有 AI 生成的内容均为**草稿参考**，必须经业务方（管理者 / HR / 面试官）**人工核对并二次确认**后才能正式提交。AI 不做最终决策，仅降低业务人员的心智负担和重复劳动。

### 1.2 架构策略

采用 **"嵌入式微能力 + 独立 AI 工作台 + AI 管理中心"** 的三层产品形态：

- **嵌入式微能力**（6 个，面向业务用户）：AI 能力直接嵌入到业务流程中，在"写评语"、"定薪资"、"投递简历"等真实操作节点主动出现，不需要用户主动去"AI 菜单"调用。统一走 `invokeAIAbility()` API，留痕进 AI 调用记录表
- **独立 AI 工作台**（1 个，面向重度业务用户）：能力本身就是一个完整的工作场景（如批量匹配 N 个简历到 1 个职位），需要独立页面承载多 Tab、规则配置、结果对比等。**走独立 API 模块**（如 `aiMatching`），不接入 `invokeAIAbility()` 留痕（详见 §4.6）
- **AI 管理中心**（面向 HR 管理员）：集中展示嵌入式能力的目录、开关、调用记录审计，位于「数据洞察 → AI 辅助中心」（`/insight/ai-assistant`）。**注意**：独立工作台模块（如智能简历匹配）有自己的统计与历史，不在此中心展示

### 1.3 交互模式分层

7 个能力按交互复杂度分为 4 类：

| 模式 | 适用场景 | 典型能力 |
|---|---|---|
| **改写型（Popover）** | 已有文本 → AI 优化 → 原地替换 | 评语优化 |
| **诊断型（Dialog/Drawer 诊断模式）** | 有上下文 → AI 分析 → 出报告只读 | OKR 质量检查、高风险预警、智能简历解析 |
| **生成型（Dialog 生成模式 / 直填）** | 上下文已知 → AI 生成 → 填入字段或复制 | 面谈提纲、薪酬检查 |
| **工作台型（独立页面）** | 多对多匹配 / 多 Tab / 规则配置 | 智能简历匹配 |

### 1.4 安全与合规

所有 AI 调用**完整留痕**：

- 调用人 / 目标对象 / 输入摘要 / 输出摘要 / 时间戳
- 采纳结果（原样采纳 / 修改后采纳 / 放弃 / 草稿未提交）
- 修改程度（0~100%，反映 AI 原稿被人工改写的比例）
- 支持按能力 / 操作人 / 目标对象 / 采纳率维度审计

### 1.5 视觉规范

**AI 入口图标全局统一**（V2.0）：

- 所有 AI 入口按钮和 AI 标识统一使用 **4 角 sparkle SVG 图标**（项目内组件名 `ArtAiIcon`，对齐 GitHub Copilot / Google Gemini / 飞书 / 钉钉等国际事实标准）
- **禁止使用 emoji（✨/📊/📄/🤖 等）表示 AI**：B 端系统中 emoji 跨平台渲染差异大（macOS / Windows / Linux 服务器表现不一致），且不适合企业级专业感
- 下文各能力章节描述按钮文案时，**不再单独标注 emoji 或图标名**（如不写"✨ AI 优化评语"，统一写"**"AI 优化评语"** 按钮"）
- Drawer 内 AI 输出区也使用同一图标，与入口按钮视觉统一

## 三、7 个 AI 能力详细说明

### 3.1 OKR 质量检查

| 维度 | 内容 |
|---|---|
| **能力编码** | `okr_check` |
| **成熟度** | 🟢 Stable（稳定） |

#### 业务痛点
- **新员工/新主管不会写 OKR**：O 太空泛、KR 无量化指标、混入 to-do list
- **Leader 没时间逐条审核**：下属交来的 OKR 质量参差不齐，Leader 每人花 10 分钟审 5 个 KR，团队 20 人 = 半天时间
- **公司 OKR 文化落地难**：SMART 原则写在培训材料里，真实写作时又用不上

#### 面向人群
- **员工**：提交 OKR 前自检（主动使用场景）
- **Leader / 直属上级**：审核下属 OKR 时快速筛查质量（被动辅助）

#### 触发入口
- **位置**：目标管理（`/perf/goal`）→ 新建/编辑目标弹窗 → KR 列表下方 **"AI 质量检查"** 按钮
- **时机**：员工写完 O 和至少 1 条 KR（关键结果文本 ≥ 10 字）后**主动点击**触发
- **交互形态**：右侧 **Drawer 抽屉**，与编辑弹窗并存（避免双层模态），打开即自动调用，结果展示后用户可逐条采纳改进版

> 🎯 **AI 边界**：AI 严格只在"事后审查"阶段介入。写 OKR 全过程（输入 O、写 KR、填权重、滑进度）**完全无 AI 干扰**——OKR 的核心价值是员工自主思考阶段性目标，AI 不替员工思考，只在员工思考成型后帮其审视规范性。

#### AI 能力类型
- **规则模型**：基于 SMART 原则（Specific / Measurable / Achievable / Relevant / Time-bound）做项目匹配
- **文本诊断**：LLM 理解 O 和 KR 的关键词、量化程度、主动性，综合评分

#### 实现规则
1. **输入**：目标标题 O + 3~5 条 KR（每条含 关键结果描述 / 度量方式 / 进度 / 权重）
   - 度量方式有 3 种：**数值进度** / **里程碑** / **达成型**——不同类型的 KR 由 AI 区分对待
2. **AI 处理**：
   - 判断 O 的"业务价值清晰度"（模糊词检测：如"提升"、"加强"、"完善"）
   - 判断 KR 的"可度量性"（数值 / 时限 / 对比基线）
   - 判断 KR 与 O 的"相关性"
   - 输出综合评分（0~5）+ 5 维度雷达（S/M/A/R/T）+ 问题清单 + 改进版草案
3. **输出（Drawer 内分两段展示）**：
   - **📊 SMART 评分卡**：综合分 + 5 维度（具体/可衡量/可实现/相关/有时限）+ 问题清单（每条标注 O / KR1 / KR2…）
   - **💡 AI 改进版**：改写后的 O + 改写后的每条 KR，提供"应用到目标 / 应用到 KR-N / 全部应用"三种采纳粒度——**用户决定要不要采纳**，AI 不强加

#### 输入/输出示例

**输入**：
```
O：提升产品质量
KR1：加强测试
KR2：修复 bug
KR3：协助领导完成相关工作
```

**输出**：
```
整体评估：整体 OKR 偏空泛，缺乏可衡量指标（SMART 得分 2.0/5）

问题分析：
• O 目标过于笼统，缺少可度量维度
• KR1"加强测试"无量化指标
• KR2"修复 bug"未设定数量/时限
• KR3"协助相关工作"偏 to-do，不构成 KR

改进建议：
O：Q2 内将产品线上缺陷率降低 40%
KR1：上线后 72 小时内缺陷数从 35 个降至 20 个以内
KR2：核心模块单元测试覆盖率从 65% 提升到 85%
KR3：建立缺陷复盘机制，每月输出 1 份分析报告
```

#### 关键约束
- **AI 不介入 OKR 写作过程**：员工填写 O / KR / 度量方式 / 进度时，**没有任何 AI 提示、AI 整理、AI 模板推荐**。OKR 的灵魂是"员工自主思考阶段性目标"，AI 替员工思考会损害这个价值。AI 只在员工写完后、主动点击 **"AI 质量检查"** 按钮时才介入。
- **员工 KR 文本完全自由**：员工可以写"测试效率降低 50%"、"完成 V2.0 上线"、"通过 PMP 认证"——任何形式都可以，不强制"指标名 + 数字字段"分工，不强制量化。AI 不评判员工写法是否"规范"。
- **采纳权在员工**：AI 改进版可逐条采纳或全部应用，**用户随时可取消**。AI 给参考、不给定论。
- **不作绩效依据**：SMART 评分仅辅助员工自检，**不进入考核数据**——避免员工为"刷高分"写 AI 喜欢的 OKR，损害真实思考。
- **触发门槛**：至少 1 条 KR 文本不少于 10 字才能调用 AI（避免随手写两个字调用 AI 的偷懒模式）。

#### 市面对标
- Lattice AI Assist
- 北森 AI 助手（OKR 模块）
- 15Five

#### 关联业务规则 · 已批准期 OKR 修订流程

> 📌 这一段不属于 AI 能力本身，而是 OKR 业务模块对**已批准 OKR 何时允许修改**的强约束。AI 质量检查（本能力）的输入 OKR 来自 OKR 业务模块，因此修订流程会影响何时可触发 AI 检查。完整业务规则放在 OKR 模块代码（`src/types/performanceGoal.ts` + `src/views/performance/goal/index.vue`），此处仅做说明。

##### 业务痛点

OKR 一旦审批通过（`approvalStatus = 1`），如果允许员工/HR 自由修改 O / KR 描述，目标管理就失效——员工本季 KR 是"完成率 80%"，做到一半发现做不到，直接改成 60%，目标的约束力被打破。

##### 业界做法对比

| 工具 | 审批前 | 已批准期 | 评估期 |
|---|---|---|---|
| **飞书 OKR** | 自由编辑 | O/KR 描述锁 → "修订"按钮 → 填修改原因 + 走审批 → 通过生效 | 冻结 |
| **钉钉 OKR** | 自由编辑 | KR 描述变更需审批 | 冻结 |
| **Lattice OKRs** | 自由编辑 | "Amend goal" 流程 | 冻结 |
| **15Five** | 自由编辑 | 软锁：可改但留痕 + 通知 manager | 冻结 |
| **本系统** | 自由编辑 | ⭐ **完整修订流程**（对标飞书/钉钉/Lattice）| 冻结 |

##### 业务规则

| 规则 | 内容 | 业界依据 |
|---|---|---|
| **R1 锁定字段** | 已批准期 `goalTitle / goalDescription / weight / keyResults` 锁定，**不能直接编辑**。进度字段（`progress / currentValue`）不锁——员工每周 Check-in 自然要改 | 飞书 / 钉钉 |
| **R2 修订流程** | 锁定后改 → "申请修订"按钮 → 填**修订原因**（必选下拉）+ 修改后内容 → 提交走审批 | 飞书 / 钉钉 / Lattice |
| **R3 同上级审批** | 修订审批人 = 初次 OKR 审批人（自动从 `PerformanceGoal.approverId` 继承）| 飞书默认 |
| **R4 时间窗口** | 季度内任意时间可申请（不限制窗口）| 业界默认 |
| **R5 修订原因必填** | 4 选 1：市场变化 / 战略调整 / 资源不足 / 其他。选"其他"时**原因说明必填** | 飞书 |
| **R6 次数限制** | **同一周期最多 1 次修订**（业界默认无限制，本系统选择保守策略）| 本系统决策 |
| **R7 修订历史留痕** | 所有修订（含已驳回）落 `revisions` 数组 + `goalRevisionMock` 表，OKR 详情页"修订历史"按钮可查 | 飞书 / Lattice |
| **R8 驳回不计数** | 驳回保持原状，`revisionCount` 不变，员工可重新申请 | 业界共识 |

##### 反面参照（不锁定的后果）

- 员工本季 KR 是"完成率 80%"，做到一半改成"60%"——绩效考核失去基线
- 直属上级看到的 OKR 永远是"刚刚改过的版本"，无法做季度复盘
- HR 看到的全公司目标聚合数据失去可比性
- 业界共识：**OKR 锁定 + 审批修订是目标管理的最低保障**

##### 与 AI 质量检查（本 AI 能力）的关联

- 修订申请 Dialog 内**不集成 AI 检查按钮**（避免 HR 提交修订前先反复跑 AI，浪费 token）
- 修订生效后，员工可在 OKR 列表对修订后的 OKR 重新点 **"AI 质量检查"** 按钮

### 3.2 员工资料卡生成

| 维度 | 内容 |
|---|---|
| **能力编码** | `meeting_agenda`（保留 code 不变；显示名为"员工资料卡生成"）|
| **成熟度** | 🟢 Stable |

#### 业务痛点
- **Leader 准备 1on1 数据分散**：要翻 OKR 系统、360 评估、出勤记录、上次纪要 4~5 个系统，耗时 30 分钟
- **关键信号容易遗漏**：单看每个数据点没什么，但"O 进度落后 + 协作分降 + 加班增"叠加起来才是协作卡点信号
- **新 Leader 不知道关注什么**：刚晋升的 Leader 不知道员工现状里哪些异常值得 1on1 时聊

#### 面向人群
- **直属 Leader**（1on1 场景）
- **评估人 / 直属上级**（绩效面谈场景）
- **HR**（面谈监督时的参考）

#### 触发入口

两处，都是 **右侧 Drawer 抽屉模式**（仅参考查看，不填入业务字段）：

1. **1on1 管理**（`/perf/one-on-one`）→ 发起 1on1 弹窗 → "准备内容" textarea **下方**链接 **"查看员工资料卡（AI 辅助参考）"**
   - 交互：**右侧 Drawer 抽屉**（540px 宽），与发起 1on1 弹窗**并存**，避免双层模态
   - Drawer 内只读展示资料卡 + "复制 / 关闭"两个按钮，**不填入业务字段**
   - 前置条件：必须先选好员工
   - "准备内容"字段：**Leader 自己写**——AI 资料卡只是辅助 Leader 了解员工现状，**怎么聊**仍由 Leader 决定

2. **绩效面谈**（`/perf/meeting`）→ 填写纪要弹窗 → 顶部 **"查看员工资料卡（AI 辅助）"** 按钮
   - 交互：同款右侧 Drawer 抽屉
   - 不污染纪要数据（纪要是面谈**后**写的，资料卡是面谈**前**看的）

**Drawer 内部结构**：
- 顶部：员工档案数据（默认折叠，可展开看 RAG 原始数据）
- 中部：AI 资料卡内容（数据快照 + AI 智能洞察 + 重点关注 三段式）
- 底部：橙色提示"本卡仅供参考，1on1 怎么聊由 Leader 决定" + 复制/关闭按钮

> 🎯 **AI 边界**：AI 把"数据整理"和"对话设计"这两件性质不同的任务**严格分开**。
>
> | 任务 | 谁做 |
> |---|---|
> | 从员工档案拉 OKR / 360 / 出勤数据 | ✅ AI（节省 Leader 翻 5 个系统的时间）|
> | 跨维度异常关联（"O 落后 + 协作降 + 加班增" → "协作卡点"）| ✅ AI（规则引擎做不到） |
> | 决定 1on1 怎么开场、先聊什么、怎么提问 | ❌ **Leader 自己**（这是领导艺术，AI 不替代）|

#### AI 能力类型

参考 Workday Insights / Lattice People Intelligence 业界共识，**规则层 + LLM 层**：

| 层次 | 谁做 | 任务 |
|---|---|---|
| **第 1 层 · 异常检测** | 规则引擎（5~10 条阈值规则）| OKR 进度 < 50% 标 ⚠️ ；360 评分降 > 0.3 标 ⚠️ ；加班 +20% 标 ⚠️ ；上次议题文本相似度 > 70% 标 🔁 |
| **第 2 层 · 组合解读** | LLM | 看到多个 ⚠️ 推断业务含义；优先级排序；自然语言总结；识别延续性议题 |

**规则层穷举不可能**（5 维度 × 各阈值 × 组合数 = 上百条），所以**必须用 LLM 做组合解读**。

#### 实现规则

**1. 输入：系统通过 RAG 自动拼接员工真实数据**（非用户手动输入）：

```
员工：{ 姓名 + 职级 + 部门 }

【本季度 OKR 进度】（来自 OKR 系统）
  逐条列出每个 O 和完成度

【360 评分趋势】（来自 360 评估，已脱敏汇总，不显示具体评估人）
  各维度上轮 vs 本轮均值，标注 ↑↓
  自评 vs 他评的差距

【近期信号】（来自出勤系统、代码仓库）
  加班时长环比变化
  代码 commit 数环比变化
  缺勤异常

【上次面谈记录】（来自历史面谈）
  时间距今多久
  议题（如"跨团队沟通改进"）
  解决度（自评）
```

**2. AI 处理（两层）**：

- 规则层先标注异常（每个数据点 ⚠️ / 🔁）
- LLM 层基于标注组合做解读

**3. AI 输出 · 三段式资料卡**：

- **📊 数据快照**：脱敏汇总的真实数据（OKR / 360 / 出勤 / 上次议题）
- **🤖 AI 智能洞察**：跨维度关联推断的"业务含义"，不超过 5 行
- **📋 重点关注**：按优先级排序的关注点（不是"建议提问"）

**4. 严格不输出"建议提问 / 讨论框架 / 对话节奏"**——这些是 Leader 的领导艺术，AI 不替代。

#### 输入/输出示例

**输入**（系统通过 RAG 自动拼接，**真实数据**，非编造）：

```
员工：张伟（P6 后端工程师 · 技术部）

【本季度 OKR 进度】
  O1 [推动核心系统重构] 75%
  O2 [跨团队协作改进] 40%
  O3 [技术分享与团队建设] 90%

【360 评分趋势】（脱敏汇总，不显示具体评估人）
  协作维度：上轮 4.2 → 本轮 3.7（↓ 0.5）
  技术维度：上轮 4.5 → 本轮 4.6（↑ 0.1）
  自评 vs 他评差：协作维度自评高 0.6 分

【近期信号】
  近 4 周加班时长：+30%（环比）
  近 2 周代码 commit：-40%（环比）

【上次 1on1】
  时间：3 个月前
  议题：跨团队沟通改进（已解决度：50%）
```

**输出 · 资料卡格式（推荐）**：

```
张伟 · 1on1 资料卡

📊 数据快照
  · OKR 进度：O1 75% / O2 40% ⚠️ / O3 90%
  · 360 趋势：协作 4.2→3.7（↓） / 技术 4.5→4.6（↑）
  · 自评高于他评 0.6 分（协作维度）⚠️
  · 近 4 周加班 +30%，但 commit -40% ⚠️
  · 上次 1on1 议题"跨团队沟通"3 个月前 🔁 延续

🤖 AI 智能洞察
  ▸ O2 进度落后 + 协作分下降 + 加班增但产出降——
    可能存在跨团队协作卡点（耗时不出活）
  ▸ 上次 1on1 已谈过跨团队沟通，本次属于延续性议题
  ▸ 协作维度自评高于他评 0.6 分，存在自我认知偏差
  ▸ 技术能力评分仍在上升（亮点）

📋 重点关注（按优先级）
  1. O2 协作卡点（高优先级 · 多信号叠加）
  2. 协作分下降的具体场景
  3. 自评与他评的协作认知偏差
  4. 技术成长方向（亮点延伸）

—— 数据来源：OKR 系统 / 360 评估（已脱敏）/ 出勤记录 / 代码仓库
—— 本卡仅供 Leader 了解员工现状，1on1 具体怎么聊由 Leader 决定
```

> 注意输出有 3 段：**数据快照**（来源真实，规则层标注）+ **AI 智能洞察**（LLM 跨维度解读）+ **重点关注**（优先级排序）。

---

**反面示例**（❌ **错误的输出**，违反"AI 不做对话设计"约束）：

```
建议讨论要点：
1. 询问员工 KR2 的具体卡点是什么
2. 询问员工最近是否觉得压力大
3. 建议先肯定技术能力的提升
4. 提问"你觉得自己在哪些方面还可以做得更好？"
```

**为什么错**：这些都是**对话设计**——怎么开场、怎么提问、怎么引导——是 Leader 的领导艺术，AI 不该替代。Leader 看资料卡知道"员工有协作卡点"就够了，**怎么聊是 Leader 的事**。

#### 关键约束

1. **🔴 P0 · AI 不替 Leader 设计对话**（最高约束）
   - AI 输出必须是**数据洞察**，不能是**对话建议**
   - 严禁出现"建议询问 X" / "可以这样开场" / "提问方向" 这种内容
   - Leader 看资料卡 → 自己决定怎么聊
   - 违反此规则 = AI 越界做领导艺术，损害 Leader-员工对话价值

2. **🔴 P0 · 严格不编造数据**
   - 所有数据点必须来自 RAG 检索的真实业务数据
   - 数据缺失就显示"暂无"，**不允许 LLM 生成"看起来真实"的虚假数据**
   - 真接入 LLM 时必须有后处理校验（详见下方）

3. **🟡 P1 · 360 数据严格脱敏**
   - 输出只显示**维度均值变化**（如"协作 4.2 → 3.7"）
   - **不显示具体评估人**或评估人数量（避免 Leader 反推匿名评估人身份）
   - 360 评估的匿名机制是合规底线

4. **🟡 P1 · 不填入业务字段**
   - 1on1 入口：AI 输出**不填入**"准备内容"——Leader 必须自己写
   - 绩效面谈入口：同样**不填入**纪要字段
   - 资料卡只是参考，不是替代品

5. **🟢 P2 · 重点关注 ≠ 必须聊**
   - AI 给的优先级是**数据信号强度**，不是"必须聊的话题"
   - Leader 完全可以选择"这次先不聊数据异常，先聊员工成长"——这是 Leader 的判断

> 真接入 LLM 时的工程实现细节（Step 1-N / Prompt 模板 / 后处理校验）
> 详见独立文档 [`docs/HR系统-AI能力实现指南.md`](./HR系统-AI能力实现指南.md) § 1

#### 市面对标

**资料卡 / 数据洞察派**（推荐路径）：
- **Workday Insights** —— 规则引擎 + LLM 自然语言；输出"员工健康度数据卡"，不写对话建议
- **Lattice People Intelligence** —— 信号标注 + LLM narrative；用于人才盘点准备

### 3.3 高风险员工预警

| 维度 | 内容 |
|---|---|
| **能力编码** | `risk_alert` |
| **成熟度** | 🟡 Beta |
| **业界对标** | Lattice Employee Health Score（首选）+ Visier At Risk Score（次选）|
| **反面参照** | 深信服离职倾向分析（2022-02 下架）+ Microsoft Productivity Score（2020-11 回缩）|

#### 业务痛点
- **HR 人力有限**：员工 200+ 人，HR 不可能每周逐一检查每个人的状态
- **风险信号分散**：可能离职的员工，OKR 更新停滞 / 360 协作分下滑 / 薪酬偏离中位 / 直属上级变更——单个信号都不明显，叠加起来才值得关注
- **发现往往滞后**：等员工提离职 HR 才反应过来，已经来不及关怀

#### 面向人群
- **HR / HRBP**（主用户）：月度风险盘点 + 盘点会议前关注名单生成
- **直线经理**（次用户）：1on1 准备前查看下属风险信号
- **不是什么**：不是日常监控工具、不是绩效淘汰依据、不是裁员名单生成器

#### 触发入口

**单一入口** —— **人才发展 → 人才盘点**（`/talent/review`）→ 页面顶部橙色 AI 洞察 Banner → **"识别高风险员工"** 按钮

> ⚠️ **为什么单一入口**（不在数据洞察 / 组织人员分析等其他页面冗余设置）：
> - 业界 5 个主流产品（Lattice / Workday / Visier / Glint / 北森）**全部是单一入口**
> - "数据洞察 → 组织人员分析"原本是看**组织数据图表**（司龄分布 / 学历分布 / 年龄分布），AI 风险预警是强行塞入
> - 真实 HR 工作流：**识别风险员工是月度/季度任务，不是日常浏览数据时的顺手动作**
> - 人才盘点是更天然的工作流嵌入位置（盘点会议前必看），与 Workday Talent Insights 嵌入 Talent Review Meeting 的设计哲学一致

> ⚠️ **路径变更说明**：人才盘点在 V2.0 Phase 7 重构（2026-04-24）中，已从原"绩效管理 → 人才盘点"（`/perf/talent-review`）拆出，独立挂在新一级菜单"人才发展"下。视图文件物理位置仍在 `src/views/performance/talent-review/index.vue`（仅路由迁移），用户感知到的菜单路径已变。

##### 交互形态

**右侧 Drawer 抽屉**（800px 宽，复用 OKR / 简历解析同款体验）：

- **顶部合规标注条**（橙黄底）：明示数据源 + 不监控行为数据 + 不做心理诊断 + 仅作 HR 关怀参考
- **扫描范围卡**：扫描人数 / 数据窗口 / 快照时间 / 信号源声明（折叠展开）
- **风险摘要统计**：扫描人数 / 高风险数 / 中风险数 / 低风险数（4 块统计卡）
- **主区分两栏**：
  - 左侧 260px：员工列表（按风险等级 + 信号数量排序），每行显示姓名 + 部门 + 风险标签 + 已跟进/关注中/已忽略状态标签
  - 右侧 flex: 1：选中员工详情
    - 详情 Header：姓名 + 部门 + 司龄 + 风险等级
    - **AI 一段话观察**（indigo 强调 + ArtAiIcon 图标）：≤ 80 字，**仅描述事实，不下结论 / 不建议干预 / 不做心理诊断**
    - **关键风险因素（Top 3 Key Drivers）**：标签云，浓缩"为什么这位员工被标"
    - **信号变化清单**（Visier "Variance" 视图）：每条 = 信号名 + current ← baseline + 变化值 + 时间窗口 + 数据源 + 规则层 flag（normal / attention / concerning）
    - **4 个 Inline 操作按钮**（详见下方"操作链路"段）

- **首次使用合规声明 Modal**（强制阻断）：未点"我已知悉"前不能查看任何数据
- **底部提示**：风险预警仅作 HR 关怀触发参考，**不可作为考核 / 裁员依据**

#### AI 能力类型

| 层次 | 谁做 | 任务 |
|---|---|---|
| **L0 · 信号采集** | HR 各业务子系统（OKR / 360 / 薪酬 / 出勤）| 每天 03:00 批处理一次，落库"风险快照" |
| **规则层 · 多信号异常检测** | 规则引擎（不是 LLM）| 每个信号独立判断 normal / attention / concerning；多信号叠加判定风险等级 high / medium / low |
| **L2 · AI 自然语言观察** | 轻量 LLM | 把规则层输出翻译成业务语言（≤ 80 字），**仅摆事实，不下结论** |

> 🎯 **AI 边界**：
>
> | 任务 | 谁做 |
> |---|---|
> | 信号变化值计算（OKR 更新频率、360 分趋势）| ✅ 规则层（数学计算）|
> | 多信号叠加 → 风险等级（high/medium/low）| ✅ 规则层（不让 LLM 判定）|
> | AI 一段话观察（描述发生了什么）| ✅ 轻量 LLM（输入 = 规则层输出）|
> | "建议邀约面试 / 推荐 1on1" 等干预决策 | ❌ **HR 自己**（AI 不替 HR 做决策）|
> | 心理诊断（"心理波动 / 抑郁倾向"）| ❌ **完全禁止**（合规雷区）|
> | 绩效下滑 / 是否裁员判定 | ❌ **不在本能力**（绩效有专门低绩效流程）|

#### 实现规则

##### 1. 信号源（**仅 HR 系统已授权数据 · 7 类**）

✅ **本系统采用的 7 类信号**：

| # | 信号名 | 数据源 | 计算示例 |
|---|---|---|---|
| 1 | 目标更新频率 | OKR 系统 | OKR 推荐节奏 = **每周/双周 Check-in**（业界共识：飞书 OKR / 钉钉 OKR / Lattice / 15Five）。判定：**连续 ≥ 3 周无 Check-in** = concerning |
| 2 | 360 评价趋势 | 360 评估 | 本季 vs 历史均值变化值（4.2 → 3.7，↓0.5）|
| 3 | 司龄 | 员工档案 | 当前司龄文本（"3 年 8 个月"）|
| 4 | 薪酬竞争力 | 薪酬系统 | 同岗位同级分位（P25 / P50 / P75）|
| 5 | 直属上级变更 | 员工档案 | 汇报关系变更日期（< 3 月 = 适应期）|
| 6 | 出勤异常 | 出勤系统 | 近 N 周请假频率 vs 历史均值 |
| 7 | 晋升停滞 | 绩效历史 | 连续 N 个考核周期未晋升 |

❌ **明确不纳入的信号**（业界共识 · 合规雷区）：

| 砍掉信号 | 原因 |
|---|---|
| **协作活跃度（IM / 会议 / 代码 review 频率）** | 行为监控雷区。Microsoft Productivity Score 2020-11 因此被舆论击退；深信服 2022-02 离职倾向分析因可查"招聘网站访问记录"24 小时内下架。Lattice / Workday / Visier / Glint 主动选择不读 IM/邮件/代码 |
| **加班时长** | "下降 60%" 含义模糊（可能是项目正常调整 / 公司反内卷导致），易误判 |
| **自评 vs 他评差异** | 容易诱导 LLM 输出"心理波动"标签，合规风险 |

##### 2. 风险等级判定（规则层叠加，**不让 LLM 判定**）

| 等级 | 判定规则（示例） | UI 标签颜色 |
|---|---|---|
| 🔴 **高风险** | ≥ 3 个 concerning 级信号 | danger |
| 🟡 **中风险** | 2 个 concerning 或 ≥ 3 个 attention 级信号 | warning |
| 🟢 **低风险** | 1 个 attention 级信号 | info |

##### 3. 输出结构（`RiskAlertResult` 类型，参见 `src/types/performanceAI.ts`）

```typescript
{
  scope: string                  // "技术研发中心全员（共 48 人）"
  dataWindow: string             // "近 8 周（2026-03-06 ~ 2026-04-30）"
  dataSources: string[]          // 信号源声明（合规追溯，UI 顶部展示）
  employees: RiskAlertEmployee[] // 员工列表（按风险等级排序）
  meta: {
    totalScanned: number
    highRiskCount, mediumRiskCount, lowRiskCount: number
    snapshotTime: string         // "2026-04-30 03:00:00"（每天 03:00 批处理）
    lastRecomputeReason: 'daily_03' | 'meeting_created' | 'manual_init'
    nextScheduledRun: string     // "2026-05-01 03:00:00"
  }
}

RiskAlertEmployee {
  employeeId, name, department, level, tenure
  riskLevel: 'high' | 'medium' | 'low'  // 仅"近期可能离职"单一目标，不下子分类
  signalChanges: RiskSignalChange[]      // Variance 视图清单
  keyDrivers: string[]                   // Top 3 关键风险因素（≤ 20 字/条）
  aiObservation: string                  // ≤ 80 字 · 仅摆事实
  followUpStatus?: 'pending' | 'reviewed' | 'snoozed' | 'in_focus' | 'one_on_one_scheduled'
  snoozedUntil?: string
}
```

##### 4. 数据流转 · 触发机制

**核心哲学**：批处理刷新 + HR 主动 pull + 1 个 push 例外（业界对标 Workday Talent Insights 24h 模式）。HR **不会被定时通知打扰**——风险预警是"长期趋势"判断而非"实时探针"，过度通知会引发疲劳。

###### 4.1 自动触发（仅 1 个时机 + 1 个 push 例外）

| # | 时机 | 频率 | 行为 | 是否打扰 HR |
|---|---|---|---|---|
| ① | **每天 03:00** 全量批处理 | 365 次/年 | 重算所有员工的 信号清单 + 风险等级 + LLM 一段话观察 → 落库 `risk_alert_snapshot` | ❌ 不打扰 · 静默落库 |
| ② | **HR 创建盘点会议时** 立刻跑 | ≈ 4 次/年 | 同 ①，但 `lastRecomputeReason = 'meeting_created'` + Banner 红点提示"已就绪" | ⚠️ **唯一 push 例外**（HR 自己发起的工作流，不算骚扰）|

**为什么是每天 03:00**：
- 业界 Workday 是每 24h 全量批处理 —— **当前 LLM 成本（GPT-4o-mini ~ $91/年/500 员工）已经不是阻碍**
- 每天跑 → HR 看到的最旧 1 天前 → **心智简单**："我点开看就是新鲜的"

**为什么不需要"业务节点驱动"**（OKR 季度结束 / 360 评估出结果等）：
- 反正第二天 03:00 会跑 → 节点触发跑是冗余
- 仅保留「创建盘点会议」push 例外 —— HR 此刻不愿等到第二天

###### 4.2 HR 主动触发（仅 1 个场景）

| 场景 | 操作 | 行为 |
|---|---|---|
| HR 月度巡检 / 1on1 准备 / 盘点会议筹备前 | 进 `/talent/review` → 点 Banner "识别高风险员工" | 读最新快照（**非实时调 LLM**）|

**为什么不提供"重新分析"按钮**：在每天跑的模式下，HR 永远看到不超过 1 天的数据，不需要手动重算。提供该按钮反而会让 HR 反复点浪费 LLM 调用。

###### 4.4 触发权限

| 角色 | 可触发查看范围 |
|---|---|
| **HR / HRBP** | 全公司员工的风险预警 |
| **直线经理** | 仅自己直属下属的风险预警（递归 N 层 ≤ 2） |
| **高层管理者** | 自己管辖部门的员工 |
| **员工本人** | ❌ 不可见（业界共识，避免自证预言）|

###### 4.5 快照陈旧告警

虽然每天跑，但定时任务跪了或数据库延迟仍可能出现陈旧快照。Drawer 顶部展示新鲜度标签：

| 距今 | 标签颜色 | 含义 |
|---|---|---|
| ≤ 1 天 | 🟢 success | "今日生成" / "1 天前" |
| 2~7 天 | 🟡 warning | 提示 HR 数据可能陈旧 |
| > 7 天 | 🔴 danger | 严重过期，HR 应联系运维排查定时任务 |

###### 4.6 与现有系统其他模块的差异说明

**重要**：现有系统其他模块（OKR / 360 / 1on1 / 审批等）**全部是事件驱动**——员工提交、HR 审批、流程节点切换。**风险预警是首次引入"定时任务"运维模式**。

| 模块 | 触发模式 |
|---|---|
| OKR / 360 评估 | 事件驱动（员工提交、周期切换）|
| 1on1 / 盘点会议 | 事件驱动（HR 创建）|
| **风险预警**（本能力）| ⭐ **批处理 + 事件混合**（每天 03:00 cron + 创建盘点会议触发）|

引入定时任务的合理性：
- 业界主流产品（Workday / Lattice / Visier 等）均有定时任务
- 风险预警**本质是批处理分析**（多信号跨员工聚合），不是事件驱动场景
- mock 阶段不实现 cron，文档明示"真接入后由后端 cron 跑"

###### 4.7 数据存储评估

| 项 | 数值 |
|---|---|
| 单条 `RiskAlertEmployee` 快照 | ≈ 4KB JSON（含信号清单 + AI 观察 + Key Drivers）|
| 全员快照 | 500 员工 × 4KB = 2MB/次 |
| 全年快照量 | 365 次 × 2MB ≈ 730MB |
| 3 年回溯（合规要求）| ≈ 2.2GB |
| 加索引 | ≈ 4GB |

✅ 完全可控，单表零头级别。

**反面示例**（❌ **错误的输出**，违反"禁止心理诊断 / 禁止行为监控 / 禁止 AI 越界决策"约束）：

```

        · 加班时长下降 60%                    ← 含义模糊已砍
        · 360 自评与他评差异显著              ← 诱导心理标签已砍
  建议：HRBP 介入关怀，必要时引入 EAP          ← AI 越界做心理干预决策

```

**为什么错**：
- "离职倾向 + 心理波动" 子分类标签 = AI 越界做心理诊断（《精神卫生法》第 23 条）
- "内部沟通活跃度（IM/会议/代码）" 监控 = 行为数据红线（参考 Microsoft Productivity Score 2020 / 深信服 2022 反面案例）
- "建议引入 EAP" = AI 替 HR 做心理干预决策，越界
- "预测准确度 92%" = LLM 自评置信度不可靠
- 应改为 **L1 信号变化清单 + L2 AI 一段话观察 + 规则层风险等级**，HR 自己下结论

#### 操作链路

Drawer 详情区底部提供 4 个 Inline 操作（**全部真功能，不是占位**）：

| 按钮 | 功能 | 实现 | 落地 |
|---|---|---|---|
| **发起 1on1** | 跳转 1on1 页面 + Query 参数预填员工信息，自动打开新建 Dialog | `router.push({ path: '/perf/one-on-one', query: { createForEmployeeId, createForEmployeeName, createForDepartment } })` | localStorage `hr_risk_alert_followups` 记 `one_on_one_scheduled` |
| **标记已跟进** | HR 完成关怀后标注，下次打开 Drawer 显示"已跟进"标签 | localStorage `hr_risk_alert_followups` 记 `reviewed` | 不入审计库（HR 个人级别） |
| **加入关注列表** | 持续跟踪某员工，列在 HR 个人收藏 | localStorage `hr_risk_focus_list` 数组 | 与简历收藏架构一致 |
| **暂时忽略 30 天** | 已确认非真实风险（如员工正在做大块技术债），下次扫描跳过该员工直到截止时间 | localStorage `hr_risk_alert_snooze` `{empId: deadlineISO}` | Drawer 渲染时过滤已 Snooze 员工 |

> 🎯 **4 个操作均为 HR 个人级别**（localStorage），不互通、不入数据库、不进 AICallRecord 调用记录审计——这些是浏览偏好与跟进状态，与 AI 输出本身无关。AI 调用本身（点"识别高风险员工"）会被 AICallRecord 留痕。

#### 关键约束

1. **🔴 P0 · 严禁心理诊断标签**
   - 不允许 "心理波动 / 抑郁倾向 / 情绪异常 / 心理问题" 等心理类标签
   - 《精神卫生法》第 23 条：精神障碍诊断仅由专业医师执行
   - 后处理校验拦截关键词：`心理波动 / 抑郁 / 情绪异常 / 心理异常 / EAP / 心理干预 / 心理问题`

2. **🔴 P0 · 严禁行为数据监控**
   - 不读 IM / 邮件 / 代码 commit / 会议时长 / 浏览器历史
   - 信号源**仅限**：OKR 系统 / 360 评估 / 员工档案 / 薪酬系统 / 出勤系统 / 绩效历史
   - 反面参照：Microsoft Productivity Score（2020-11 因此被强制聚合化）+ 深信服离职倾向分析（2022-02 24 小时内下架）

3. **🔴 P0 · AI 不下子分类标签**
   - 单一预测目标：仅"未来 4~8 周内主动离职可能性"
   - 不下"绩效下滑 / 离职倾向 / 心理波动" 等子分类
   - 不外延到"是否邀约面试 / 是否调薪 / 是否裁员"等决策

4. **🔴 P0 · AI 一段话观察严禁建议性动词**
   - aiObservation 字段 ≤ 80 字
   - 后处理拦截："建议 / 推荐 / 应该 / 必须 / 建议邀约 / 建议干预"
   - 仅描述事实（"近 8 周该员工 360 协作分由 4.2 降至 3.7..."）

5. **🟡 P1 · HR 必须人工复核才能行动**
   - AI 输出仅作 HR 关怀触发参考
   - 4 个 Inline 操作（发起 1on1 / 已跟进 / 加入关注 / 暂时忽略）由 HR 主动触发，AI 不自动执行

6. **🟡 P1 · 不能作为任何考核 / 裁员依据**
   - Drawer 底部明示
   - AICallRecord 记录任何对外引用本能力输出做考核决策的违规行为

7. **🟡 P1 · 员工不可见自身风险**
   - 默认仅 HR / 经理可见
   - 但**首次使用本能力**前 HR 必须看合规声明 Modal（localStorage `hr_risk_alert_acknowledged_v2` = '1' 才放行）
   - 系统首页"员工告知"中说明"系统会基于 HR 数据做风险分析"（《个保法》第 24 条要求）

8. **🟢 P2 · 批处理刷新（非实时 LLM 调用）**
   - **每天 03:00** 自动跑，落库"风险快照"（业界对标 Workday 24h 模式）
   - HR 创建盘点会议时立刻跑（push 例外）
   - HR 触发"识别高风险员工"实际是读快照（800ms 延迟来自 mock，真接入后是缓存读取）
   - 详细数据流转见 §3.3「实现规则 · 数据流转 · 触发机制」段

### 3.4 评语优化

| 维度 | 内容 |
|---|---|
| **能力编码** | `comment_polish` |
| **成熟度** | 🟢 Stable |

#### 业务痛点
- **管理者表达能力参差**：有的写"还不错"过于口语化，有的过于直白甚至失礼
- **国内员工对"评语"敏感**：管理者措辞不当容易引起误解或不满（"为什么说我'差'？"）
- **专业书面表达耗时**：每人想 15 分钟才能写出一段语气专业的评语，10 个下属 = 2.5 小时
- **管理者思考价值不可替代**：评语承载的是 manager-employee 反馈对话，**核心内容必须由管理者自己产出**，AI 不能代写

#### 面向人群
- **直属上级**（写下属评语主场景）
- **HR**（查看历史评语时的 AI 演示查看）

#### 触发入口
两处：
1. **评估人视角**：绩效评估（`/perf/evaluation`）→ 填写评分弹窗 → "整体评语" textarea 下方 **"AI 优化评语"** 链接
   - 交互：**Popover 小浮层**（420px 宽），打开即自动调用，输出展示在原文下方
   - 顶部带 **3 个风格切换按钮**（专业 / 简洁 / 建设性），切换后重新调用 AI
   - 底部带"放弃 / 替换原文"两个按钮
2. **HR 视角**：绩效评估 → 查看评估详情 → 每个节点评语旁 **"AI 优化（演示）"** 按钮
   - 交互：Dialog **生成模式** + `adoptable=false`（只预览+复制，不替换原数据）

#### AI 能力类型
- **风格润色**：口语化 → 书面化、主观词 → 客观词、模糊词 → 准确词
- **风格切换**：3 选 1（专业 / 简洁 / 建设性），改变语气但不改变内容
- ❌ **不做事实补充**：AI 不查看员工 OKR / 360 / 项目数据，**不增加任何原稿没有的细节**
- ❌ **不做结构化重写**：不强制转 SBI / GROW 等结构（管理者觉得需要可手动选模板，但 AI 不主动）

> 🎯 **设计原因**：评语是管理者对员工的真实反馈，内容必须管理者自己产出。AI "补事实" 在 Mock 阶段尚可演示，**真接入 LLM 时会编造数据**（员工档案里出现"QPS 2000 提升到 2800"等虚构信息），这是产品事故级风险。业界 Lattice / Quantive / SAP / 北森的评语 AI 均**仅做风格润色**，不补事实。

#### 实现规则
1. **输入**：管理者写好的评语初稿（≥ 5 字，否则 AI 不触发）
2. **AI 处理**（严格规则）：
   - 提取原稿的核心观点和评价倾向（褒/贬/中性）
   - 仅做"口语化 → 书面化"的语气切换，按用户选定风格输出
   - **保持原稿评价倾向不变**（褒就是褒，贬就是贬，AI 不洗白也不加重）
   - **保持原稿事实集合不变**（输入提到的事实 = 输出能用的事实）
   - 输出长度 ≤ 输入长度 × 2
3. **输出**：润色后的评语（与输入字数同量级，无新增事实）

#### 输入/输出示例

**通用输入**：
```
张伟本季度表现还不错，完成了主要目标，但沟通方面需要加强
```

**输出 · 风格 1 · 专业**（默认）：
```
张伟本季度整体表现良好，已达成本职核心目标。在沟通协作方面仍有
提升空间，建议进一步加强主动同步与跨团队对齐。
```
> 特点：书面化、规范客观；保留原意（褒贬程度不变）；无新增事实

**输出 · 风格 2 · 简洁**：
```
张伟本季度核心目标达成，沟通协作待加强。
```
> 特点：极简，提炼最关键判断；适合评估表单字段紧凑场景；无新增事实

**输出 · 风格 3 · 建设性**：
```
张伟本季度整体表现良好，已达成核心目标，值得肯定。沟通协作方面
仍有提升空间，期待下季度在跨团队主动同步上有更进一步的表现。
```
> 特点：在原稿基础上加 1 句温和的期待；适合年终评估等需要正向引导的场景；无新增事实

---

**反面示例**（❌ 错误的 AI 输出，**违反"不补事实"约束**）：

```
张伟本季度核心目标达成率 95%，其中主导的缓存架构升级项目（KR1）
超额 15% 完成，QPS 从 2000 提升到 2800……
```

**为什么错**：原稿没说"达成率 95%"、"缓存架构升级项目"、"KR1"、"超额 15%"、"QPS 2000 提升到 2800"——**这些是 AI 编造**。如果管理者没核对就采纳，员工档案会出现虚构信息。

> ⚠️ 真接入 LLM 时，输出层后处理校验**必须拒绝**这种含编造事实的输出（详见下方"防编造工程方案"）。

#### 关键约束

1. **🔴 P0 · 严格不补事实**（最高约束）
   - AI 不允许添加原稿没有的具体数据：数字、百分比、项目名、人名、时间点、KR 编号、职级
   - 真接入 LLM 时必须有后处理校验，**违反此规则的输出直接拒绝并重试**
   - 这是产品事故级红线：违反 = 员工档案出现虚构信息

2. **🟡 P1 · 保留原稿评价倾向**
   - 原稿是褒义（"还不错、表现好"）→ AI 输出仍是褒义
   - 原稿是贬义（"沟通需加强、未达预期"）→ AI 输出不能洗白成褒义
   - 原稿是中性 → AI 不能加重为褒或贬

3. **🟡 P1 · 输出长度可控**
   - 输出字数 ≤ 输入字数 × 2
   - 避免 AI 过度扩写凑字数（这反而是编造的伪装）

4. **🟢 P2 · 用户随时可放弃**
   - Popover 底部"放弃"按钮始终可点
   - 已替换原文的内容用户仍可编辑

5. **🟢 P2 · 触发门槛**
   - 原稿至少 5 字才能调用 AI（避免空文本调用）

> 真接入 LLM 时的工程实现细节（Step 1-N / Prompt 模板 / 后处理校验）
> 详见独立文档 [`docs/HR系统-AI能力实现指南.md`](./HR系统-AI能力实现指南.md) § 3

- **Lattice AI Writing Assist** —— 4 个风格按钮（Friendly / Concise / Professional / Make it shorter），不补事实

### 3.5 智能简历解析

| 维度 | 内容 |
|---|---|
| **能力编码** | `resume_parse` |
| **成熟度** | 🟢 Stable |

#### 业务痛点
- **投递量大**：一个热门岗位每天 50+ 份简历，HR 一天 8 小时全看都看不完
- **关键字段提取耗时**：HR 要扫一遍简历找"学历 / 工作经历 / 项目 / 技能 / 期望薪资"
- **读不出"这个人"**：单纯 NER 抽取只回答"简历里写了什么"，回答不了"这个人稳不稳定 / 主擅长什么 / 行业聚焦在哪"
- **HR 个人筛选信息丢失**：HR 浏览简历池时心里记下的"这份不错"，关掉页面就丢了

#### 面向人群
- **HR / 招聘官**（主要使用者）
- **用人经理**（查看候选人详情时辅助判断）

#### 触发入口
- **位置**：简历池（`/recruit/resume`）→ 列表"详情"→ 跳转候选人详情独立页（`/recruit/resume/detail/:id`）→ 顶部面包屑右侧 **"AI 简历解析"** 按钮（ArtAiIcon SVG 图标 · 4 角 sparkle，**无 emoji**）
- **交互形态**：**右侧 Drawer 抽屉**（800px 宽，复用 OKR / 1on1 资料卡同款体验）
- **Drawer 内结构**（按业界共识 · Greenhouse 2024 Redesign 砍 Tab 改主区+右侧栏单页布局）：
  - **Header 区**：候选人姓名（24px）+ 当前职位@公司 + 工作年限 + 学校学历 + 期望薪资 + 5 个操作按钮（查看原文 / 邀面试 / 收藏 / 加入人才库 / 关闭）
  - **AI 一段话摘要**（顶部 indigo 强调 + ArtAiIcon 图标 + 脉冲动画）：≤ 100 字回答"这个人是谁"
  - **主区 65%**：工作经历（合并项目段）/ 教育 / 技能
  - **右侧 35%**：联系方式 / 字段校验（折叠）/ AI 衍生洞察（折叠）/ 期望薪资
  - **简历原文**：按需 Dialog（不默认占视觉）
- **不污染候选人档案**：Drawer 仅参考，HR 复核后**自行决定**是否更新候选人档案字段

> 🎯 **AI 边界**：简历解析是**信息抽取 + 衍生洞察**任务（NER + 规则校验 + L2 跨字段关联），相对客观。AI 边界限制在：
>
> | 任务 | 谁做 |
> |---|---|
> | 从原始简历抽取结构化字段（L1） | ✅ AI（NER 是机器擅长的）|
> | 字段抽取的规则校验（"日期完整吗"、"姓名为空吗"） | ✅ 规则层（不让 AI 自评置信度）|
> | L2 衍生洞察（平均在职时长 / 技能使用上下文 / 跳槽频率 / 行业聚焦度 / 一句话总结）| ✅ 规则层 + 轻量 LLM（基于 L1 结构化数据，非凭空生成）|
> | 与目标岗位的批量匹配度排序 | ❌ **不在本能力**（归 §3.7 智能简历匹配）|
> | 决定"是否邀约面试" | ❌ **HR 自己**（AI 不替 HR 做录用决策）|

> 📌 **与 §3.7 智能简历匹配的边界**：本能力解决"读懂这一个人"，§3.7 解决"在 50 个候选人里挑 5 个"。**本能力不输出与岗位匹配度评分**，避免与 §3.7 重复——HR 用本能力把简历读懂，再去 §3.7 工作台看批量打分排序。

#### AI 能力类型

| 层次 | 谁做 | 任务 |
|---|---|---|
| **L1 · NER 信息抽取** | LLM（基础能力）| 从简历文本提取基本信息 / 教育 / 工作经历 / 项目 / 技能 / 期望薪资 |
| **规则层 · 字段校验** | 规则引擎 | 字段级独立校验（"至今"是否歧义 / 日期是否完整 / 必填是否为空），**非 LLM 自评置信度** |
| **L2 · AI 衍生洞察** | 规则 + 轻量 LLM | 基于 L1 结构化数据做跨字段关联：平均在职时长 / 技能使用上下文 / 跳槽频率 / 行业聚焦度 / AI 一段话总结 |

#### 实现规则

1. **输入**：候选人简历原文（OCR 后的纯文本）
2. **L1 · AI 抽取 6 类字段**（NER 基础能力）：
   - 基本信息：姓名 / 年龄 / 出生 / 性别 / 电话 / 邮箱 / 所在地
   - 教育经历（多段）：学校 / 专业 / 学历 / 起止时间
   - 工作经历（多段，按时间倒序）：公司 / 部门 / 岗位 / 起止时间 / 时长 / 业绩条目 / 关键词标签
   - 项目经历（多段）：名称 / 角色 / 时长 / 团队规模 / 产出 / 技术栈
   - 技能：主技能（高频）/ 加分技能 / 方向标签
   - 期望薪资：月薪 / 年包 / 估算年薪
3. **规则层校验**（每个字段独立判断 ok / warning）：
   - ✓ 字段完整且格式规范
   - ⚠️ 字段缺失或格式歧义（如"至今" / "year 18" / 工作经历跳期）
4. **L2 · 衍生洞察**（基于 L1 结构化数据计算，非凭空生成）：
   - **平均在职时长**：基于 `workExperience.duration` 计算（如"约 2.5 年/段"）—— 职业稳定性指标
   - **技能使用上下文**（**不是单纯频次**）：每条 = 技能 + 时长 + 真实使用场景（来自 `workExperience.achievements` / `projects.output`），如 `Golang · 5 年 · 抖音中台 / 美团交易系统`
   - **跳槽频率**：低 / 正常 / 高（基于平均在职时长 + 段数判定）+ 文字解释
   - **行业聚焦度**：基于 `workExperience.keywords` 聚类
   - **AI 一段话总结**：≤ 100 字，回答"这个人是谁"
5. **输出**：L1 字段抽取 + 规则层 fieldChecks + L2 insights（结构化，前端按 Header / 摘要 / 主区 / 右侧栏 渲染）

**反面示例**（❌ **错误的输出**，违反"禁止伪精度 / 禁止 AI 越界录用决策 / 禁止与岗位匹配度"约束）：

```
解析结果（置信度 92%）：              ← LLM 自评置信度不可靠
技能匹配：9.1 / 10                  ← 9.1 这个精度的依据是什么？
推荐邀面试：是（综合分 8.5）          ← AI 不替 HR 做录用决策
匹配 P7 强匹配（命中 5/6 项）         ← 本能力不输出与岗位匹配度（归 §3.7）
```

**为什么错**：
- "92% 整体置信度" 是 LLM 自我报告偏差，学术界已证明不可靠
- "9.1 / 10" 这种小数一位精度暗示"非常精确的算法"，但实际是 LLM 拍脑袋编的
- "推荐邀面试" 是 HR 决策权，AI 越界
- "匹配 P7" 与 §3.7 边界重复，本能力不再做岗位匹配度
- 应改为**字段级规则校验**（✓ / ⚠️）+ **定性洞察**（稳定性良好 / 跳槽频率低）+ **明确命中清单**

#### 操作链路

Drawer 顶部 Header 区右侧提供 4 个操作按钮（**全部真功能，不是占位**）：

| 按钮 | 功能 | 实现 |
|---|---|---|
| **查看原文** | 弹 Dialog 显示完整简历 OCR 文本 | 按需触发，不默认占视觉 |
| **邀面试** | 跳转面试创建页，预填候选人信息 | 跳 `/recruit/interview/create/new` |
| **收藏** | HR 个人偏好（localStorage `hr_starred_resumes`），列表行星标 + 简历池筛选「仅看我的收藏」与之联动 | 不入库、不写候选人档案 |
| **加入人才库** | 调 `talentProfileApi.add` + `linkResumeToTalent`，跳转人才档案 | 真创建 `TalentProfile`，**不需"复核"中间态**（系统中无审批队列） |

> 🎯 **收藏 = HR 个人级别**：
> - 每个 HR 看到的星标不互通；不入库、不进 `AICallRecord` 调用记录
> - HR 浏览简历池时心里记下的"这份不错"由前端 localStorage 持久化
> - **三处实时同步**（单一来源 = localStorage）：列表行星标 ↔ Drawer 收藏按钮 ↔ 简历池筛选「仅看我的收藏」

#### 关键约束

1. **🔴 P0 · 严禁伪精度数字**
   - 不允许 "9.1 / 10"、"匹配度 8.5" 这种小数精度——LLM 拍脑袋编的
   - 改为**定性档位**（强/中/弱、低/正常/高）+ **明确命中清单**
   - 真接入 LLM 时，后处理校验**强制拦截输出含"X.Y / 10"格式**

2. **🔴 P0 · 严禁 LLM 自评置信度**
   - 不允许 "解析结果（置信度 92%）"
   - 改为**字段级规则校验**（每个字段独立 ok / warning）
   - 校验规则由**后端规则层**判断（如"日期格式合法"、"姓名是否为空"），不让 AI 自评

3. **🔴 P0 · L2 衍生洞察严禁凭空编造**
   - "平均在职时长 3.5 年" 必须基于 L1 `workExperience.duration` 计算
   - "技能使用上下文" 的 `scenarios` 字段必须来自 L1 `workExperience.achievements` 或 `projects.output`
   - 真接入 LLM 时：L2 prompt 必须以 L1 JSON 为输入，**禁止 LLM 引用任何 L1 之外的字段**

4. **🟡 P1 · 不输出与岗位匹配度评分**
   - 本能力**专注"读懂这个人"**，与岗位匹配度排序归 §3.7
   - Drawer 内不显示"匹配 P7 强匹配"、"司龄符合"这类分维度评分
   - `ResumeParseResult` 类型已**不再包含** `matchAnalysis` 字段（参见 `src/types/performanceAI.ts`）

5. **🟡 P1 · HR 必须人工复核才能入库**
   - AI 输出仅作初筛参考
   - "加入人才库" 按钮直接创建 `TalentProfile` + 关联简历，但**不直接覆盖**候选人档案的现有字段
   - Drawer 仅展示，由 HR 自行决定是否更新

6. **🟡 P1 · 收藏不进 AICallRecord 审计**
   - 收藏是 HR 浏览偏好，与 AI 输出无关
   - localStorage `hr_starred_resumes` 单一来源

### 3.6 薪酬合理性检查

| 维度 | 内容 |
|---|---|
| **能力编码** | `salary_sanity_check` |
| **成熟度** | 🟡 Beta |
| **业界对标** | ChartHop Compensation Reviews + Workday Compensation + Pave Visual Offer Letter |

#### 业务痛点

定薪 / 调薪环节是 HR 工作中**风险最高 + 信息最不对称**的场景之一：

- **定薪依赖 HR 经验**：新 HR 不知道各岗级的带宽，容易定偏
- **倒挂难发现**：给下属涨薪时经常忘了核对是否超过同岗位老员工或团队 Leader
- **同岗公平性缺监控**：同岗同级员工薪资差距逐渐拉大，HR 无感知
- **审批决策缺数据**：审批人看到的是 Offer 表单，不知道这个数字是否合理

#### 面向人群
- **HR / 招聘官**（Offer 定薪）
- **HR / HRBP**（发起调薪申请）
- **审批人**（用人经理 / HRD，审批时看到诊断结果作为参考）

#### 触发入口

两处，都采用**薪资输入框下方 inline 提示**模式（业界主流形态，对齐 ChartHop / Pave）：

1. **Offer 定薪**：招聘 → Offer 管理 → 创建 Offer 表单 → 薪资输入框下方
2. **调薪申请**：薪酬福利 → 调薪管理 → 新增申请表单 → "调薪后薪资"输入框下方

交互：
- 薪资填齐后，**默认 2000ms debounce 自动触发**（避免 HR 反复试探薪资浪费 LLM 调用）
- 用户也可以通过"AI 诊断"按钮**手动提前触发**
- 输出：状态 Tag + Compa-Ratio + 同岗分位 + AI 一句话观察 + "查看完整诊断"
- 完整诊断 800px **Drawer**（与其他 AI 能力 Drawer 形态统一，appendToBody 与外层 Offer Dialog 不冲突）：顶部状态 alert（含 AI 一句话副标题）+ 带宽可视化（min/mid/max + 提议位置点）+ 同岗对标 + 数值倒挂提示 + 合规底部

#### AI 能力类型

⚠️ **重要边界澄清**（业界共识 · ChartHop / Workday / Pave 一致）：

| 检测项 | 实际实现 |
|---|---|
| 超带宽检查 | 🟢 **规则引擎**（数学：薪资 vs min/max）|
| 数值倒挂检测 | 🟢 **规则引擎**（数学：A vs B 比大小）|
| 同岗内部对标 | 🟢 **规则引擎 + 检索**（SQL 查同岗薪酬分布算分位）|
| Compa-Ratio 计算 | 🟢 **数学**（实发 / 中位数）|
| 自然语言一句话观察 | 🟡 **AI 辅助说明** |
| 异常模式检测 | 🟡 **AI 辅助** |

**关键事实**：业界把"超带宽红/黄/绿"包装成"AI 检测"是**营销，不是技术差异**。这件事的核心是规则引擎，AI 仅"翻译数据"。

> 🎯 **AI 边界**：
>
> | 任务 | 谁做 |
> |---|---|
> | 带宽对比 / 分位计算 / 倒挂检测 | ✅ 规则引擎（不是 AI）|
> | 把规则结果用自然语言陈述（"提议月薪 32k 超带宽上限 7%..."）| ✅ 轻量 LLM |
> | 给"高/中/低风险等级"主观判断 | ❌ **不允许**（业界共识 = 客观二元提示）|
> | 给"方案 A/B/C 推荐" | ❌ **不允许**（业界明确反对，AI 越界做决策）|
> | "建议晋升 P5→P6"等组合建议 | ❌ **不允许**（跨越薪酬域到人才决策）|
> | 暴露具体他人薪资数字 | ❌ **不允许**（用分位 P25/P50/P75 表达）|
> | 下"管理倒挂"判断 | ❌ **不允许**（专家序列等场景下数值倒挂可能合理）|

#### 实现规则

##### 1. 输出结构（`SalarySanityCheckResult`）

```typescript
{
  status: 'in_range' | 'out_of_range' | 'inversion_alert'  // 二元 + 倒挂提示
  band: {              // 带宽分析（规则层）
    min, mid, max, proposed: number
    compaRatio: number   // 提议 / 中位
    outOfRange: boolean
    source: string       // 数据来源（公司薪酬系统 + 周期）
  }
  peer: {              // 同岗内部对标（规则层 · 不暴露具体他人薪资）
    peerCount: number
    proposedQuartile: 'P25' | 'P50' | 'P75' | 'Top'
    source: string
  }
  inversion: {         // 数值倒挂提示（事实陈述，不下管理判断）
    hasInversion: boolean
    description: string    // 如"超出团队同岗 75 分位"
    source: string
  }
  aiObservation: string  // ≤ 60 字 · 仅描述事实
}
```

##### 2. 三档状态（不是高/中/低主观等级）

| status | 含义 | 触发条件 |
|---|---|---|
| `in_range` 🟢 | 在带宽内 | min ≤ 提议 ≤ max 且无倒挂 |
| `out_of_range` 🟡 | 超出带宽 | 提议 > max 或 提议 < min |
| `inversion_alert` 🔴 | 形成数值倒挂 | 提议 > 团队同岗 P75 |

##### 3. Compa-Ratio 颜色分级（业界标准）

| 区间 | 含义 | 颜色 |
|---|---|---|
| 0.85 ~ 1.10 | 正常 | 🟢 |
| 1.10 ~ 1.25 | 偏高 | 🟡 |
| > 1.25 | 高风险 | 🔴 |
| < 0.85 | 偏低 | ⚪ |

#### 输入/输出示例

**输入**（系统自动构造）：
```
员工：李某（P5 前端工程师，司龄 3.2 年，当前月薪 25k）
调整方案：调薪至 月薪 32k（涨幅 28%）
调整原因：本年度绩效 S 档，主导完成 2 个核心项目
```

**输出**（结构化，前端按 inline 卡片 + 完整 Dialog 渲染）：

```
状态：out_of_range（超出带宽）

【带宽分析】
  P5 前端带宽：min 22k / mid 26k / max 30k
  提议薪资：32k
  Compa-Ratio：1.23（偏高）
  数据来源：公司薪酬系统 · 前端 P5 带宽（更新 2026-Q1）

【同岗内部对标】
  同岗级（P5 前端）在职 12 人 · 提议薪资处于 Top 分位
  数据来源：公司薪酬系统 · 同岗级在职员工分布
  （注：不展示具体他人薪资数字）

【数值倒挂提示】
  提议薪资超出团队同岗 75 分位
  数据来源：公司薪酬系统 · 直属团队同岗薪资分布
  （注：仅事实陈述，不下"管理倒挂"判断）

【AI 观察】（≤ 60 字 · 仅描述事实）
  提议月薪 32k 超出前端 P5 带宽上限 30k 约 7%。Compa-Ratio 1.23（中位 26k）。同岗级 12 人中处于 Top 分位。

—— AI 诊断结果仅作 HR 决策参考，不可作为唯一依据
```

**反面示例**（❌ **错误的输出**，违反业界共识）：

```
合理性诊断：⚠️ 偏高，建议调整         ← "建议调整"是 AI 越界做决策

【倒挂风险】
  ⚠️ 该员工直属 Leader 月薪 30k       ← 暴露他人具体薪资数字
  调整后构成管理倒挂                   ← AI 不下"管理倒挂"判断（专家序列下可能合理）

【调整建议】（按优先级）                 ← AI 不给方案 A/B/C
  方案 A（推荐）：同步晋升 P5→P6       ← 跨越薪酬域到人才决策
  方案 B：保留 P5，薪资调至 29k        ← AI 替 HR 决策
  方案 C（当前方案）：保留 P5，直接给 32k
```

**为什么错**：
- "建议调整 / 方案推荐" = AI 替 HR 决策，业界共识明确反对
- "Leader 月薪 30k" = 暴露他人具体薪资，违反《个保法》第 28 条 + 公司薪酬保密制度
- "构成管理倒挂" = AI 下主观判断，专家序列等场景下数值倒挂可能合理
- "建议晋升" = 跨越薪酬域，AI 越界

#### 关键约束

1. **🔴 P0 · 严禁 AI 给"方案推荐"或"建议晋升"**
   - 业界（Workday / ChartHop / Pave）共识：AI 仅做"输入"，不做"决策者"
   - 后处理校验拦截关键词：`建议晋升 / 推荐方案 / 方案 A / 方案 B / 方案 C / 建议调整 / 应该调整`

2. **🔴 P0 · 严禁暴露具体他人薪资数字**
   - 用分位（P25/P50/P75/Top）替代具体数字
   - 后处理校验拦截"该员工直属 Leader 月薪 X 元"等格式
   - 合规依据：《个保法》第 28 条（薪酬属敏感信息）+ 公司薪酬保密制度

3. **🔴 P0 · 严禁下"管理倒挂"判断**
   - AI 仅陈述"形成数值倒挂"事实
   - 不输出"构成管理倒挂"等主观结论
   - 专家序列 / 历史包袱 / 特殊保留等场景下数值倒挂可能合理，HR 自行判断

4. **🟡 P1 · 数据来源必须标注**
   - 每条诊断结果（带宽 / 分位 / 倒挂）必须标注 `source` 字段
   - 真接入 LLM 时：薪酬带宽数据**必须从公司薪酬系统读取**，LLM **不允许编造**

5. **🟡 P1 · 状态用"二元"，不用"主观等级"**
   - 不输出"高/中/低风险等级"标签
   - 用 `in_range / out_of_range / inversion_alert` 三档客观状态
   - Compa-Ratio 颜色分级是数学规则，不是主观判断

6. **🟡 P1 · 非阻塞 + 非阻碍填表**
   - 诊断卡不阻止用户提交薪资，只是信息展示
   - debounce 默认 2000ms 避免 HR 反复试探浪费 LLM 调用
   - HR 也可手动点"AI 诊断"按钮提前触发

7. **🟢 P2 · 摘要 + 详情**
   - inline 状态卡只显示 Compa-Ratio + 分位 + AI 一句话
   - 完整诊断需点"查看完整诊断"打开 **Drawer**（800px · appendToBody · 顶部 alert 含 AI 一句话 + 带宽可视化 + 同岗对标 + 倒挂提示 + 合规底部）

#### 关联业务规则 · Offer 流程（业界共识对齐）

> 📌 这一段不属于 AI 能力本身，而是 AI 嵌入的业务环境。AI 薪酬合理性检查的**主入口在 Offer 创建表单**，因此 Offer 流程的设计直接影响 AI 能力的工作效果。

##### Offer 流程的 4 条业务规则

| 规则 | 内容 | 业界依据 |
|---|---|---|
| **R1 必须候选人驱动** | Offer 必须从候选人详情页（面试通过状态）发起，不允许凭空创建 | Greenhouse / Workday / Lever / iCIMS / Moka / 北森**全部强制** |
| **R2 列表页定位 = 进度看板 + 批量操作** | Offer 列表页**不是创建入口**，是 HR Leader 看进度 + 批量审批 + 报表导出的工作台 | Greenhouse Offer Activity Report 模式 |
| **R3 列设计按 Tab 差异化** | 进行中 Tab 显示「Compa-Ratio / 状态停留 / 预计入职」识别卡住的 Offer；已结束 Tab 砍掉这三列，改显示「结束时间」（按状态取 expectedJoinDate / responseTime / approveTime / updateTime） | 业界 Greenhouse 实践 |
| **R5 删除按钮按状态条件渲染** | 仅"审批拒绝/候选人拒绝/已撤回"可删，其他状态删除按钮**直接不显示**（业界 Greenhouse / Workday 共识：无效操作不显示比 disabled 更干净）| Greenhouse 删除策略 |
| **R6 列表前端过滤分页** | 一次拉全量数据（pageSize=10000），前端做 Tab 过滤 + 搜索过滤 + 状态过滤 + 分页（避免后端分页与前端 Tab 过滤错位）| Greenhouse / Workday 通用模式 |
| **R7 状态筛选按 Tab 收窄** | 筛选区"状态"下拉的选项**按 Tab 自动收窄**：进行中 Tab 显示 1/2/4 三项；已结束 Tab 显示 3/5/6/7 四项。Tab 切换时自动清空已选 status 避免错位 | Greenhouse / Moka 一致 |
| **R8 列宽规范** | 所有列用像素 `min-width`（业界中后台标准），操作列用固定 `width="240" fixed="right"`。表格 `style="width: 100%"`。长文本列（岗位 / 部门）加 `show-overflow-tooltip` | CLAUDE.md 表格列宽规则 |
| **R9 Tab 与按钮布局** | Tab 单独一行（在 data-card 内 status-tabs 区）；批量删除按钮在 table-header **左侧**（`header-buttons`），与本系统其他页面（如 interview / talent-pool）保持一致 | 本系统统一规范 |
| **R10 状态名（Offer 模块）** | 状态 2 显示文字 = "审批通过"（不是"已批准"，避免与 OKR 模块"已批准"语义混淆）；状态 7 显示文字 = "已撤回"（不是"已失效"）| 飞书审批 / 钉钉一致 |
| **R4 状态机 7 个状态闭环** | 草稿 → 待审批 → 审批通过/驳回 → 已发送 → 候选人接受/拒绝 → 已撤回 | 业界标准 |

##### 为什么 Offer 必须候选人驱动？

1. **候选人是 Offer 的唯一合法源头** —— 没有 candidate 的 Offer 是数据孤儿，无法关联简历 / scorecards / 面试评估
2. **审计合规要求溯源** —— 法务/合规审计要回答"这个数字怎么来的"，Job → Candidate → Offer 是审计证据链
3. **角色责任分工** —— Recruiter 对 Offer 负责，Coordinator 对 Job/Candidate 负责，Hiring Manager 对面试评估负责
4. **业界纪律** —— Greenhouse 社区原话："让 offer 创建多几次点击，是预防大错的廉价代价"

- **入职部门预填**：从候选人详情发起 Offer 时，调 `getJobPostingDetail(jobId)` 拿 `departmentName` + `workLocation` 一起传 query 预填（候选人 Resume 没有 department 字段，需从 JobPosting 查）
- **创建 Offer 弹窗字段权限**（业界对标 Greenhouse / Workday / Lever / Moka）：
  - 候选人姓名 / 入职岗位 / 入职部门 = **只读 disabled**（来自候选人 profile，HR 不可改）
  - HR 可编辑 = 薪资 / 试用期 / 入职日期 / 工作地点 / 福利 / 汇报对象 / Offer 模板 / 其他说明
- **批量删除状态校验**（业界标准）：
  - ✅ 可删：草稿（无）/ 审批拒绝 / 候选人拒绝 / 已撤回
  - ❌ **绝对不可删**：待审批 / 审批通过 / **已发送 / 已接受**（法务证据）
  - 选中含不可删状态时弹错误提示，列出违规 Offer 编号

❌ **已砍**：
- Offer 列表"+ 创建 Offer"按钮（业界铁律：所有主流 ATS 都不支持凭空创建）
- 简历池搜索（CandidatePicker）—— 凭空创建已砍，搜索失去意义
- 顶部引导 alert —— 业界（Greenhouse / Workday / Lever / Moka / 北森）**没一家**有这种 alert，候选人驱动的 Offer 模式 HR 自然能理解列表页定位

##### 未来演进方向（不在本次范围）

- **超带宽自动升级审批**：Greenhouse Offer Exception / Lever Conditional Step 标杆做法 —— 提议薪资超带宽时自动加塞高级别审批人。**前提是结构化薪酬带宽数据 + 审批配置中心支持条件路由**，本次未做

- **Offer 接受 → 自动创建员工档案**：业界做法是 hire 业务流自动转换。本次未核查我们系统的链路完整性，作为单独任务

### 3.7 智能简历匹配

| 维度 | 内容 |
|---|---|
| **能力编码** | `ai_matching`（独立模块，不在 `AIAbilityCode` 联合类型中） |
| **架构归属** | 🔵 **独立 AI 工作台**（详见 §4.6） |
| **成熟度** | 🟢 Stable（稳定） |

#### 业务痛点
- **简历库利用率低**：HR 每次开新职位都从最新投递逐一过简历，公司过去攒下的几千份历史简历几乎不被翻查
- **岗位推荐靠人脑记忆**：HR 看到一份优质简历，要凭记忆判断"我现在哪个 JD 适合 ta"，新 HR 完全不知道公司有哪些坑位
- **筛选标准不一致**：每个 HR 对"什么算强匹配"理解不同，同一份简历不同 HR 给的优先级可能差 2 档
- **缺批量评估能力**：单份简历的解析（§3.5）只能告诉你"这个人怎么样"，回答不了"这一批 50 个候选人哪几个最该面试"

#### 与"智能简历解析（§3.5）"的区别

| 维度 | §3.5 简历解析 | §3.7 简历匹配 |
|---|---|---|
| **方向** | 单份简历 → 结构化字段 | 多对多评估（N 简历 × M 职位） |
| **使命** | 把简历**读懂**（NER 抽取 / 字段化） | 把简历**排序**（多维评分 / 推荐等级） |
| **触发** | 候选人详情 → AI 解析 Tab（嵌入式） | 独立工作台页面（多 Tab） |
| **结果对象** | 1 个候选人的结构化档案 | N 个候选人的总分排行榜 |
| **API 路径** | `invokeAIAbility('resume_parse', ...)` | `batchMatch()` / `recommendJobs()` |

两个能力**互补不冲突**：HR 通常先用 §3.5 把简历读懂，再用 §3.7 把简历库批量打分排序。

#### 面向人群
- **HR / 招聘官**（主要使用者）：开新职位时一键扫库 / 拿到优质简历时定位最匹配的 JD
- **招聘负责人**：维护匹配规则（不同岗位族用不同权重）
- **业务用人经理**：查看匹配排行榜辅助决策（只读）

#### 触发入口

**主入口**：招聘 → 简历池（`/recruit/resume`）→ 顶部 ModuleTabBar → **"AI 智能匹配"** 标签（实际跳转隐藏路由 `/recruit/ai-matching`）

> 这是"独立工作台"模式：从简历池主页面切换 Tab 进入，不弹 Dialog，不在 form 边上 inline。

进入后是一个完整的多 Tab 工作页：

| Tab | 名称 | 功能 |
|---|---|---|
| 1 | **简历批量匹配** | 选 1 个职位 + N 份简历 + 选用规则 → AI 评分排序 |
| 2 | **职位推荐** | 选 1 份简历 + N 个在招职位 + 选用规则 → 推荐最匹配的 JD |
| 3 | **匹配规则** | 维护"岗位族 → 权重配置 + 关键词词典 + 阈值"的规则库 |

页面顶部有 4 个统计卡：累计匹配 / 强匹配数 / 一般匹配数 / 平均分。

#### AI 能力类型
- **多维度加权评分**：5 个维度（技能/经验/学历/地域/薪资）各算 0-100 分，按规则权重加权得总分
- **关键词命中检测**：从规则的"核心技能词典"匹配候选人简历，输出**命中关键词**和**缺失关键词**
- **阈值分级**：根据规则配置的强匹配/一般/待定三档阈值，把总分映射到 4 个等级
- **LLM 文本建议**：基于评分结果生成"AI 建议"自然语言（如"技能命中但经验略浅，可作为备选"）

#### 实现规则

**评分维度（5 维）**：

| 维度 | 字段 | 默认权重示例（前端岗） |
|---|---|---|
| 技能匹配 | skill | 45% |
| 经验匹配 | experience | 25% |
| 学历匹配 | education | 10% |
| 地域匹配 | location | 10% |
| 薪资匹配 | salary | 10% |

> 各岗位族的权重可在"匹配规则"Tab 自定义。如管理岗的经验权重默认到 45%，技能权重压到 20%。

**推荐等级**（基于总分 + 规则阈值）：

| 等级 | 默认阈值 | Tag 样式 | 业务含义 |
|---|---|---|---|
| 🟢 强匹配（strong_match） | 总分 ≥ 80 | success | 强烈推荐，建议直接进入面试 |
| 🔵 一般匹配（match） | 65 ≤ 总分 < 80 | primary | 可推进，重点考察缺失能力 |
| 🟡 待定（maybe） | 50 ≤ 总分 < 65 | warning | 备选，关注学习能力 |
| ⚪ 不匹配（not_match） | 总分 < 50 | info | 不建议推进 |

**匹配规则（4 条预置）**：
- 技术研发岗 · 前端方向（核心技能：Vue / React / TypeScript / JavaScript / CSS / HTML）
- 技术研发岗 · Java 方向（核心技能：Java / Spring Boot / MyBatis / MySQL / Redis）
- 产品设计岗 · 通用（核心技能：需求分析 / 产品设计 / Axure / 用户研究 / B 端 / C 端）
- 管理岗 · 总监及以上（核心技能：团队管理 / 战略规划 / 跨部门协作 / 项目管理）

每条规则可配置：维度权重、核心技能、加分技能、淘汰关键词（命中归零）、三档阈值。

**场景 A · 批量匹配（最高频）**：

6. 下方表格自动展示"匹配结果（按总分降序）"，前 3 名带 🥇🥈🥉 奖牌

#### 关键约束
- **AI 排序仅作初筛参考**：HR 必须人工核对前 N 名简历后才决定面试名单，**禁止"AI 推荐什么就面试什么"**
- **结果不进 AI 中心审计**：本能力的调用记录、命中率、采纳率走独立的"匹配历史"统计，**不出现在** `/insight/ai-assistant` 的全局调用记录里。如需审计本能力使用情况，需要在简历池工作台单独看
- **规则即产品**：匹配效果完全取决于规则配置质量。建议每个新岗位族建独立规则，避免用通用规则评所有岗
- **淘汰关键词慎用**：命中淘汰词会**直接归零**整份简历得分，配置不当可能误杀（例如把"应届"加进淘汰词会同时淘汰所有应届优秀候选人）
- **批量匹配上限**：建议单次批量评估不超过 200 份简历，避免页面卡顿

## 四、统一技术基础

### 4.1 调用链路

```
业务页面按钮触发
      ↓
组件层（AIAssistPopover / AIAssistDialog / AISalarySanityInline）
      ↓
API 层 invokeAIAbility(code, input, operator, targetEmployee?)
      ↓
Mock 层 mockInvokeAI()  →  真实环境：调用 AI 凭证（GPT-4 / Claude / 文心一言）
      ↓
留痕：写入 aiCallRecordMock（调用记录表）
      ↓
返回输出 + recordId
      ↓
用户处置（采纳 / 修改采纳 / 放弃 / 草稿）
      ↓
updateAIAdoption(recordId, result, modificationDegree)
```

### 4.2 留痕表结构（AICallRecord）

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | number | 记录 ID |
| `abilityCode` | string | 能力编码 |
| `abilityName` | string | 能力名称 |
| `operatorName / operatorId` | string / number | 调用人 |
| `targetEmployee / targetEmployeeId` | string / number | 目标对象（如员工姓名）|
| `inputSummary` | string | 输入摘要（前 100 字）|
| `outputSummary` | string | 输出摘要（前 100 字）|
| `adoptionResult` | enum | `adopted` / `modified_adopted` / `rejected` / `draft` |
| `modificationDegree` | number | 修改程度（0~100%）|
| `callTime` | string | 调用时间 |
| `createTime` | string | 创建时间 |

### 4.3 采纳率追踪指标

AI 中心提供以下效果指标（管理员视角）：

| 指标 | 计算 | 业务含义 |
|---|---|---|
| **总调用量** | COUNT(调用记录)| AI 使用活跃度 |
| **采纳率** | `(adopted + modified_adopted) / total` | 整体效果（越高越好）|
| **原样采纳率** | `adopted / total` | AI 输出质量（越高说明 AI 越准）|
| **修改采纳率** | `modified_adopted / total` | AI 作为起点价值（说明 AI 省去了从 0 到 70% 的时间）|
| **放弃率** | `rejected / total` | AI 输出偏差（高放弃率说明需要调优）|
| **平均修改程度** | AVG(modificationDegree) | 员工对 AI 输出的信任度 |

### 4.4 AI 能力管理中心

**位置**：数据洞察 → AI 辅助中心（`/insight/ai-assistant`）

**功能**：
- **能力目录**：7 张卡片展示所有能力 + 成熟度 + 适用场景
- **业务入口跳转**：每张卡片点击"跳转到业务入口"直达真实触发点
- **演示调用**：管理员可演示能力（不影响真实业务数据）
- **调用记录审计**：按能力 / 操作人 / 时间维度查询留痕

### 4.5 AI 凭证管理

**位置**：招聘配置 → 外部集成

**管理内容**：
- LLM 提供商（GPT-4 / Claude / 文心一言 / 混元 / 通义）
- API Key（加密存储）
- 每月调用配额
- 各能力的模型路由（如 OKR 检查用 GPT-4，简历解析用 Claude）

### 4.6 架构分层：嵌入式 vs 独立工作台

本系统明确区分两种 AI 能力的代码与产品架构：

| 维度 | 嵌入式微能力（§3.1~§3.6，共 6 个） | 独立 AI 工作台（§3.7，共 1 个） |
|---|---|---|
| **API 路径** | `src/api/performanceAI.ts` 的 `invokeAIAbility()` | `src/api/aiMatching.ts` 的 `batchMatch()` / `recommendJobs()` |
| **能力注册** | 在 `src/mock/performanceAI.ts` 的 `abilitiesData` 数组中（7 项） | **不注册**，独立的 `MatchingRule` + `AIMatchResult` 表 |
| **类型定义** | `src/types/performanceAI.ts` 的 `AIAbilityCode` 联合类型 | `src/types/aiMatching.ts` 独立类型体系 |
| **UI 组件** | 三件套：`AIAssistDialog` / `AIAssistPopover` / `AISalarySanityInline` | 独立页面 `views/recruitment/ai-matching/index.vue`，自行实现表格/雷达图 |
| **入口形态** | 嵌入业务弹窗 / textarea 边 / Banner | 独立页面 + 多 Tab + 顶部统计卡 |
| **留痕表** | `aiCallRecordMock`（统一调用记录）| `aiMatchResultCrudMock`（匹配历史）+ `matchingRuleCrudMock`（规则） |
| **AI 中心可见性** | ✅ 可在 `/insight/ai-assistant` 看能力卡 + 调用记录 + 采纳率 | ❌ 不可见。需要去简历池 → AI 智能匹配工作台单独看 |
| **采纳率追踪** | 4 状态闭环：草稿 / 采纳 / 修改采纳 / 放弃 | **无采纳率概念**：只统计强匹配数 / 一般匹配数 / 平均分 |

**设计原因**：

- **嵌入式**适合"小步嵌入业务流"的微能力，每次调用上下文小、输出短、要快速对比"采不采纳"
- **独立工作台**适合"重决策"的能力，输入是 N×M 的多对多组合、需要规则配置、结果是排行榜，不适合在弹窗里塞下

**未来演进方向**：

如果需要把 §3.7 智能简历匹配也接入统一审计（让 HR 总监能在 AI 中心看到"上周一共匹配了多少候选人、采纳率多少"），可以做一个**适配层**：在 `batchMatch()` 调用结束后，往 `aiCallRecordMock` 写一条 `abilityCode='ai_matching'` 的记录。但这会增加架构复杂度，目前评估**收益不足**——独立工作台已有自己的统计页，HR 总监需要这个数据时去简历池看就行。
