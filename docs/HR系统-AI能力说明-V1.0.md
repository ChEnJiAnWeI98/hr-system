## 一、设计理念

### 1.1 核心定位

> **AI 是效率工具，不是判断工具。**

所有 AI 生成的内容均为**草稿参考**，必须经业务方（管理者 / HR / 面试官）**人工核对并二次确认**后才能正式提交。AI 不做最终决策，仅降低业务人员的心智负担和重复劳动。

### 1.2 架构策略

采用 **"嵌入式为主 + AI 管理中心为辅"** 的产品形态：

- **嵌入式**（面向业务用户）：AI 能力直接嵌入到业务流程中，在"写评语"、"定薪资"、"投递简历"等真实操作节点主动出现，不需要用户主动去"AI 菜单"调用
- **AI 管理中心**（面向 HR 管理员）：集中展示所有能力、开关、调用记录审计，位于「数据洞察 → AI 辅助中心」

### 1.3 交互模式分层

7 个能力按交互复杂度分为 3 类：

| 模式 | 适用场景 | 典型能力 |
|---|---|---|
| **改写型（Popover）** | 已有文本 → AI 优化 → 原地替换 | 评语优化、反馈改写 |
| **诊断型（Dialog 诊断模式）** | 有上下文 → AI 分析 → 出报告只读 | OKR 质量检查、高风险预警 |
| **生成型（Dialog 生成模式 / 直填）** | 上下文已知 → AI 生成 → 填入字段或复制 | 面谈提纲、面谈提纲（1on1）、薪酬检查、简历解析 |

### 1.4 安全与合规

所有 AI 调用**完整留痕**：

- 调用人 / 目标对象 / 输入摘要 / 输出摘要 / 时间戳
- 采纳结果（原样采纳 / 修改后采纳 / 放弃 / 草稿未提交）
- 修改程度（0~100%，反映 AI 原稿被人工改写的比例）
- 支持按能力 / 操作人 / 目标对象 / 采纳率维度审计

## 三、7 个 AI 能力详细说明

### 3.1 OKR 质量检查

| 维度 | 内容 |
|---|---|
| **能力编码** | `okr_check` |
| **成熟度** | 🟢 Stable（稳定） |

#### 业务痛点
- **新员工/新主管不会写 OKR**：O 太空泛、KR 无量化指标、混入 to-do list
- **Leader 没时间逐条审核**：下属交来的 OKR 质量参差不齐，Leader 每人花 10 分钟审 5 个 KR，团队 20 人 = 半天时间
- **公司 OKR 文化落地难**：SMART 原则写在培训材料里，真实写作时又用不上

#### 面向人群
- **员工**：提交 OKR 前自检（主动使用场景）
- **Leader / 直属上级**：审核下属 OKR 时快速筛查质量（被动辅助）

#### 触发入口
- **位置**：目标管理（`/perf/goal`）→ 新建/编辑目标弹窗
- **时机**：填完 O 和 KR 后，点击 **"✨ AI 质量检查"** 按钮
- **交互**：Dialog **诊断模式**，打开即自动调用，输出报告只读

#### AI 能力类型
- **规则模型**：基于 SMART 原则（Specific / Measurable / Achievable / Relevant / Time-bound）做项目匹配
- **文本诊断**：LLM 理解 O 和 KR 的关键词、量化程度、主动性，综合评分

#### 实现规则
1. **输入**：目标标题 O + 3~5 条 KR（含描述/目标值/权重）
2. **AI 处理**：
   - 判断 O 的"业务价值清晰度"（模糊词检测：如"提升"、"加强"、"完善"）
   - 判断 KR 的"可度量性"（数值 / 时限 / 对比基线）
   - 判断 KR 与 O 的"相关性"
   - 输出综合评分（0~5）+ 问题分析 + 改进建议
3. **输出**：结构化诊断报告（SMART 得分 + 问题清单 + 改写示范）

#### 输入/输出示例

**输入**：
```
O：提升产品质量
KR1：加强测试
KR2：修复 bug
KR3：协助领导完成相关工作
```

**输出**：
```
整体评估：整体 OKR 偏空泛，缺乏可衡量指标（SMART 得分 2.0/5）

问题分析：
• O 目标过于笼统，缺少可度量维度
• KR1"加强测试"无量化指标
• KR2"修复 bug"未设定数量/时限
• KR3"协助相关工作"偏 to-do，不构成 KR

改进建议：
O：Q2 内将产品线上缺陷率降低 40%
KR1：上线后 72 小时内缺陷数从 35 个降至 20 个以内
KR2：核心模块单元测试覆盖率从 65% 提升到 85%
KR3：建立缺陷复盘机制，每月输出 1 份分析报告
```

#### 关键约束
- **只诊断，不替换**：AI 不会自动改写用户的 OKR，用户自己决定是否采纳建议
- **输出为草稿**：SMART 得分是 AI 的主观判断，不作为绩效考核依据

#### 市面对标
- Lattice AI Assist
- 北森 AI 助手（OKR 模块）
- 15Five

### 3.2 面谈提纲生成

| 维度 | 内容 |
|---|---|
| **能力编码** | `meeting_agenda` |
| **成熟度** | 🟢 Stable |

#### 业务痛点
- **Leader 不会准备 1on1**：每周/每月要和下属 1on1，讨论点零散，经常聊成无主题闲聊
- **绩效面谈前数据分散**：管理者要翻 OKR 进度、360 反馈、上次面谈纪要，耗时 30 分钟
- **讨论优先级乱**：把时间花在"客套"上，"真正要解决的问题"没提

#### 面向人群
- **直属 Leader**（1on1 场景）
- **评估人 / 直属上级**（绩效面谈场景）
- **HR**（面谈监督时的参考）

#### 触发入口
两处：
1. **1on1 管理**（`/perf/feedback-one-on-one`）→ 发起 1on1 弹窗 → "准备内容"旁 **"✨ AI 生成提纲"** 按钮
   - 交互：**直接调用**（无 Dialog）→ AI 结果直接填入"准备内容" textarea
2. **绩效面谈**（`/perf/meeting`）→ 填写纪要弹窗 → 顶部 **"✨ AI 生成面谈提纲（参考）"** 按钮
   - 交互：Dialog **生成模式** + `adoptable=false` → 只有"复制到剪贴板 / 关闭"

#### AI 能力类型
- **上下文检索（RAG）**：从员工数据池拉取近期目标进度、360 评分趋势、上次面谈纪要
- **文本生成**：LLM 基于检索结果按优先级组织讨论要点

#### 实现规则
1. **系统自动拼接上下文**（用户无需输入）：
   - 员工基本信息（姓名/职级/部门）
   - 近 3 个月 OKR 进度
   - 360 反馈评分趋势
   - 上次面谈核心议题
2. **AI 输出**：按优先级排序的 3~5 个讨论要点，每个要点含：
   - 话题描述
   - 为什么这次要聊（数据依据）
   - 建议提问方向
3. **输出直接可用**：面谈人不需要二次编辑即可参考使用

#### 输入/输出示例

**输入**（系统自动构造）：
```
员工：张伟（P6 后端工程师）
本季度目标：3 个 OKR，完成率 65%
360 评价：技术扎实，但协作维度分数有下降趋势
上次面谈：3 个月前，讨论过跨团队沟通问题
```

**输出**：
```
建议讨论要点（按优先级）：

1. 目标进度（重点）
   - KR2 进度仅 40%，了解具体卡点
   - 是否需要调整 KR 范围或寻求跨组支持

2. 协作反馈（敏感话题）
   - 最近 360 评价中协作分下降 0.5 分，询问是否有特定场景
   - 对比上次面谈，跨团队沟通是否有新的困扰

3. 成长诉求
   - 技术深度已达 P6 水平，了解是否对 P7 通道有规划

4. 支持需求
   - 当前工作是否需要资源/培训/辅导支持
```

#### 关键约束
- **1on1 场景**：结果直接填入 textarea，用户可编辑
- **绩效面谈场景**：仅参考，不自动写入纪要字段（因为纪要是面谈**后**填的，提纲是面谈**前**看的）

#### 市面对标
- Lattice Talent Reviews
- Culture Amp 1:1 Agendas
- 北森 AI 面谈助手

### 3.3 高风险员工预警

| 维度 | 内容 |
|---|---|
| **能力编码** | `risk_alert` |
| **成熟度** | 🟡 Beta |

#### 业务痛点
- **HR 人力有限**：员工 200+ 人，HR 不可能每周逐一检查每个人的状态
- **风险信号分散**：离职倾向的人，目标更新停滞、加班时长异常下降、360 协作分下滑——每个信号都不明显，但叠加起来很有信号价值
- **发现往往滞后**：等员工提离职 HR 才反应过来，已经来不及挽留

#### 面向人群
- **HR / HRBP**：月度风险盘点
- **管理者**：季度团队体检
- **人才盘点会议成员**：盘点前筛选关注对象

#### 触发入口
两处：
1. **组织人员分析**（`/insight/organization`）→ 顶部橙色 AI 洞察 Banner → "识别高风险员工"
2. **人才盘点**（`/perf/talent-review`）→ 顶部同样 Banner

交互：Dialog **诊断模式**（打开即调用，展示报告，只有关闭）

#### AI 能力类型
- **多信号异常检测**：结合 N 个弱信号做组合判断（机器学习分类器或规则引擎）
- **自然语言解释**：LLM 把算法输出的风险分数翻译成业务语言

#### 实现规则

扫描的 8 类信号：
1. **目标更新频率**：连续 N 周无更新 / 进度倒退
2. **360 评价趋势**：本轮评分 vs 历史均值下降 ≥ 0.5 分
3. **加班时长**：环比下降 60%+（失去工作热情信号）
4. **协作活跃度**：内部沟通频次下降（IM / 会议 / 代码 review）
5. **请假异常**：近 2 周请假频率高于历史
6. **自评 vs 他评差异**：差异显著（心理波动信号）
7. **直属上级变更近期**：汇报关系变化 < 3 个月
8. **薪酬对比**：同岗位同级分位 < 25 分位

风险分层：
- 🔴 **高风险**（3+ 强信号）：建议 HRBP 介入 1on1 面谈
- 🟡 **中风险**（2 个中等信号）：建议管理者关注，下次 1on1 主动聊
- 🟢 **低风险**：信号微弱，暂不干预

#### 输入/输出示例

**输入**（系统自动构造）：
```
输入范围：技术研发中心全员，近 8 周数据
```

**输出**：
```
识别到 3 位高风险员工：

⚠️ 张某（高风险，离职倾向）
  信号：• 目标更新连续 3 周停滞
        • 360 评价协作分下降 2 档
        • 加班时长下降 60%
  建议：尽快 1on1 面谈了解状态

⚠️ 李某（中风险，绩效下滑）
  信号：• 近 2 周目标进度倒退
        • 两个 KR 完成率已低于预期 20%
        • 但 360 评价无异常
  建议：协助梳理目标优先级，提供资源支持

⚠️ 王某（中风险，心理波动）
  信号：• 内部沟通活跃度大幅下降
        • 近期请假频率异常
        • 360 自评与他评差异显著
  建议：HRBP 介入关怀，必要时引入 EAP
```

#### 关键约束
- **不能作为任何考核/裁员依据**：仅作 HR 关怀的触发参考
- **员工不可见**：风险标签不对员工本人展示，避免心理负担
- **隐私合规**：用于监测的信号必须是员工入职时授权范围内的数据

#### 市面对标
- Workday People Analytics（Flight Risk Prediction）
- 北森人才盘点 - 流失风险预警
- Lattice People Intelligence

### 3.4 反馈改写（SBI 转换）

| 维度 | 内容 |
|---|---|
| **能力编码** | `feedback_rewrite` |
| **成熟度** | 🟢 Stable |

#### 业务痛点
- **同事互评"不知怎么写"**：很多人写不出具体事例，要么空泛（"他挺好的"），要么尖锐（"他太被动"）
- **上级写评语"主观判断为主"**：缺事实支撑，导致员工不服
- **360 评估沦为形式**：零信息反馈堆积，被评估者看不到成长建议

**SBI 模型**（Situation / Behavior / Impact）是业界公认的结构化反馈模型：
- **Situation**：具体情境
- **Behavior**：观察到的行为
- **Impact**：行为带来的影响

但大多数人不会手动应用 SBI。

#### 面向人群
- **同事互评评估人**（360 场景最大受众）
- **直属上级**（写下属评语时）

#### 触发入口
- **位置**：360 度评估（`/perf/review-360/x`）→ 填写反馈弹窗 → "改进建议" textarea 下方 **"✨ AI 改写为 SBI"** 链接
- **时机**：填完改进建议后（输入为空时按钮 disabled）
- **交互**：**Popover 小浮层**（不是全屏 Dialog）→ 自动调用 → 展示**原文 vs AI 版本对比** → "替换原文 / 放弃"

#### AI 能力类型
- **文本风格迁移**：把粗糙/尖锐的语言 → 建设性表达
- **结构化重写**：把零散描述 → 按 SBI 四段式组织
- **事实补充**：根据关键词从上下文补充具体情境（如"Q1 支付网关重构项目"）

#### 实现规则
1. **输入**：原始反馈（至少 20 字）
2. **AI 识别**：
   - 关键词（行为/态度/能力判断）
   - 语气等级（攻击性 / 陈述 / 建设性）
   - 缺失维度（情境？行为？影响？）
3. **重写规则**：
   - 补"情境（Situation）"：基于员工近期项目/场景
   - 提炼"行为（Behavior）"：从原始反馈中提取具体动作
   - 推导"影响（Impact）"：行为造成的结果
   - 给建议：最后一段建设性改进方向
4. **输出**：SBI 结构化完整版

#### 输入/输出示例

**输入**：
```
他太被动，推进不了事情，经常要别人催
```

**输出**：
```
SBI 结构化版本：

情境（Situation）：在 Q1 支付网关重构项目中

行为（Behavior）：在需要跨组推进的关键节点（如 API 规范对齐、压测协调），他较少主动牵头联系，多数情况是其他同事或我去跟进

影响（Impact）：导致 2 个关键节点延期各 2~3 天，下游团队对协作体验有微词

改进建议：未来类似跨组任务，希望能主动拉会、主动发同步周报。如果推进中遇到卡点，可以更早同步，以便团队一起想办法
```

#### 关键约束
- **Popover 内 inline 对比**：原文红色删除线 / AI 版绿色高亮，用户一眼看出差异
- **采纳后替换原文**：点击"替换原文"直接写入原 textarea
- **不自动提交**：用户可在替换后继续编辑

#### 市面对标
- Lattice AI Assist（Write Better Feedback）
- Culture Amp Feedback Coach
- SAP SuccessFactors Joule

### 3.5 评语优化

| 维度 | 内容 |
|---|---|
| **能力编码** | `comment_polish` |
| **成熟度** | 🟢 Stable |

#### 业务痛点
- **管理者写评语时间紧**：Q 末绩效期，每人 15 分钟写评语，动辄管 10 个下属 = 2.5 小时
- **评语表达同质化**：HR 看到一屋子评语长得一样"工作认真、态度端正、积极配合"
- **缺事实支撑**：员工看到评语感觉"空泛"，不服气
- **口吻不专业**：管理者本身表达能力参差，有的过于直白，有的过于模糊

#### 面向人群
- **直属上级**（写下属评语主场景）
- **HR**（查看历史评语时的 AI 演示查看）

#### 触发入口
两处：
1. **评估人视角**：绩效评估（`/perf/evaluation`）→ 填写评分弹窗 → "整体评语" textarea 下方 **"✨ AI 优化评语"** 链接
   - 交互：Popover 改写模式（原文 vs AI 对比 → 替换）
2. **HR 视角**：绩效评估 → 查看评估详情 → 每个节点评语旁 **"✨ AI 优化（演示）"** 按钮
   - 交互：Dialog **生成模式** + `adoptable=false`（只预览+复制，不替换原数据）

#### AI 能力类型
- **文本润色**：优化表达、语气
- **事实补充**：基于员工目标/项目补具体数据
- **结构化**：转 SBI 结构

#### 实现规则
1. **输入**：管理者评语初稿
2. **AI 处理**：
   - 提取初稿的核心观点
   - 从员工近期 OKR / 项目 / 360 评价中补事实依据
   - 转为"数据 + 行为 + 判断 + 建议"四段式
   - 保留管理者的原意（不篡改核心评价倾向）
3. **输出**：优化版评语（通常 2~3 倍字数于初稿）

#### 输入/输出示例

**输入**：
```
张伟本季度表现还不错，完成了主要目标，但沟通方面需要加强
```

**输出**：
```
优化版本（保留管理者原意，补充事实并转 SBI）：

张伟本季度核心目标达成率 95%，其中主导的缓存架构升级项目（KR1）超额 15% 完成，QPS 从 2000 提升到 2800，给业务带来明显价值。

沟通协作方面有提升空间：Q1 跨团队协作的 3 个关键项目中，有 2 次在推进卡点时未及时同步到相关方，导致下游团队被动等待。建议在 Q2 养成"每周跨组同步会 + 日清周报"的节奏。

综合评估：核心业绩突出，协作习惯需打磨，总体符合 P6 水平并具备向 P7 冲刺的技术底子。

—— AI 辅助优化，请管理者确认后提交
```

#### 关键约束
- **保留原意**：AI 不改变管理者的评价倾向（肯定 / 中立 / 批评）
- **添加免责声明**：输出结尾明确"AI 辅助优化，请管理者确认后提交"
- **采纳后可二次编辑**：用户采纳 AI 版本后还能继续改

- 北森 AI 评语助手

### 3.6 智能简历解析

| 维度 | 内容 |
|---|---|
| **能力编码** | `resume_parse` |
| **成熟度** | 🟢 Stable |

#### 业务痛点
- **投递量大**：一个热门岗位每天 50+ 份简历，HR 一天 8 小时全看简历都看不完
- **手动录入耗时**：HR 要把简历里的"学历 / 工作经历 / 技能"手动填到候选人档案
- **初筛粗糙**：HR 看第一眼需要判断"是否邀约面试"，靠直觉容易遗漏好候选人
- **期望薪资对比脱节**：候选人简历里写"期望 50k"，HR 要手动查是否在公司 P7 带宽内

#### 面向人群
- **HR / 招聘官**（主要使用者）
- **用人经理**（查看候选人详情时辅助判断）

#### 触发入口
- **位置**：简历池（`/recruit/resume`）→ 候选人"详情"→ 切换到 **"✨ AI 解析"** Tab
- **时机**：Tab **首次被打开时自动触发**，后续保持缓存；提供"重新解析"按钮
- **交互**：**自动触发**（无需用户点击按钮），loading 后展示结构化结果

#### AI 能力类型
- **信息抽取（NER）**：从非结构化简历文本提取命名实体
- **关系抽取**：识别"哪段工作经历对应哪家公司、哪个岗位"
- **结构化 LLM 输出**：JSON / YAML 格式的结构化字段
- **岗位匹配度打分**：对比目标岗位描述（JD）计算向量相似度

#### 实现规则
1. **输入**：候选人档案里的原始文本（工作经历 + 期望薪资 + 学历信息）
2. **AI 抽取 6 类字段**：
   - 基本信息：姓名 / 年龄 / 学历 / 院校 / 毕业年份
   - 工作经历：按段切分，每段抽取"公司 / 岗位 / 时间 / 关键词"
   - 技能标签：技术栈 / 方向 / 领域
   - 期望薪资：月薪 / 年包
3. **AI 附加分析**：
   - 岗位匹配度（技能匹配 + 司龄匹配）
   - 薪酬位置（对比公司带宽）
4. **输出**：结构化字段报告（含置信度）+ 匹配度分析 + 薪酬预警

#### 输入/输出示例

**输入**（系统自动拼接）：
```
姓名：陈某
出生：1993.08
学历：硕士，清华大学 计算机科学与技术（2015-2018）
工作经历：
  2018.07-2022.03  字节跳动  资深后端工程师
    · 负责抖音服务端中台架构设计，支撑 DAU 3 亿
    · 主导推荐系统改造，QPS 峰值 20w+，P99 < 50ms
  2022.04-至今  美团  后端技术专家
    · 外卖交易核心链路负责人，日订单 7000w
技能：Golang、MySQL、Redis、Kafka、K8s、分布式系统设计
期望薪资：月薪 50~65k，year 18
```

**输出**：
```
解析结果（置信度 92%）：

【基本信息】
  姓名：陈某
  出生：1993-08（32 岁）
  最高学历：硕士
  毕业院校：清华大学
  专业：计算机科学与技术
  毕业年份：2018

【工作经历】（共 2 段，7 年）
  1. 字节跳动  资深后端工程师  2018-07 ~ 2022-03（3 年 8 个月）
     关键词：抖音、后端中台、推荐系统、高并发
  2. 美团  后端技术专家  2022-04 ~ 至今（3 年 1 个月）
     关键词：外卖交易、核心链路、日千万订单

【技能标签】
  后端：Golang ▲ MySQL ▲ Redis ▲ Kafka ▲ K8s ▲
  方向：分布式系统设计、高并发、交易系统

【期望薪资】
  月薪：50k ~ 65k
  年包：18 薪
  估算年薪：¥108 万 ~ ¥140 万

【与岗位匹配度分析】（目标岗位：高级后端 P7）
  技能匹配：9.1 / 10
  司龄匹配：8.5 / 10（7 年经验，略超 P7 中位）
  薪酬匹配：⚠️ 期望薪资位于本公司 P7 带宽上 25% 分位，建议协商

—— AI 解析结果仅供参考，HR 需人工复核关键字段后导入候选人库
```

#### 关键约束
- **HR 必须人工核对关键字段后才能入库**（AI 提供草稿，不直接写候选人档案）
- **置信度展示**：AI 对自己的把握度透明（低置信度字段 HR 重点复核）
- **支持重新解析**：解析失败或换岗位目标时可手动重新触发

### 3.7 薪酬合理性检查

| 维度 | 内容 |
|---|---|
| **能力编码** | `salary_sanity_check` |
| **成熟度** | 🟡 Beta |

#### 业务痛点
- **定薪依赖 HR 经验**：新 HR 不知道各岗级的带宽，容易定偏
- **薪酬倒挂难发现**：给下属涨薪时经常忘了核对是否超过 Leader
- **同岗公平性缺监控**：同岗同级员工薪资差距逐渐拉大，HR 无感知
- **审批决策缺数据**：薪酬委员会审调薪时，没有结构化的对标信息

#### 面向人群
- **HR / 招聘官**（Offer 定薪）
- **HR / HRBP**（发起调薪申请）
- **薪酬委员会 / HRD**（审批决策）

#### 触发入口
两处，都采用**薪资输入框下方 inline 自动诊断**模式：
1. **Offer 定薪**：招聘与入职 → Offer 管理（`/recruit/offer`）→ 创建 Offer 弹窗 → 薪资输入框下方
2. **调薪申请**：薪酬福利 → 调薪管理（`/comp/adjustment`）→ 新增申请弹窗 → "调薪后薪资"输入框下方

交互：
- 关键字段（候选人/员工 + 岗位 + 薪资）填齐后，**debounce 600ms 自动触发**
- 输入框下方出现**状态卡**：彩色 Tag（合理/偏离/风险）+ 摘要 + "查看完整报告"
- 点击"查看完整报告"打开 Dialog 看详情

#### AI 能力类型
- **规则引擎**：岗位带宽对比、倒挂检测、分位计算
- **LLM 诊断生成**：把规则输出翻译成业务语言 + 给出多方案建议

#### 实现规则

AI 内部运行 4 类检查：

1. **带宽分析**：
   - 查目标岗位的带宽（P50 ± 20%）
   - 判断新薪资是否超出带宽（偏高 / 合理 / 偏低）

2. **同岗位对标**：
   - 查同岗级所有员工薪资分布
   - 判断新薪资位于 P25 / P50 / P75 / Top

3. **倒挂风险**：
   - 查员工直属 Leader 的薪资
   - 判断新薪资是否超 Leader（管理倒挂）

4. **调整方案建议**：
   - 方案 A：推荐方案（通常是晋升+调薪组合）
   - 方案 B：折中方案（保留岗级，薪资顶格）
   - 方案 C：当前方案（列出风险）

#### 输入/输出示例

**输入**（系统自动构造）：
```
员工：李某（P5 前端工程师，司龄 3.2 年，当前月薪 25k）
调整方案：调薪至 月薪 32k（涨幅 28%）
调整原因：本年度绩效 S 档，主导完成 2 个核心项目
```

**输出**：
```
合理性诊断：⚠️ 偏高，建议调整

【带宽分析】
  当前岗位：前端 P5，公司带宽 22k ~ 30k（P50=26k）
  调整后薪资：32k
  ➜ 已超出 P5 带宽上限 7%，需升级到 P6 带宽才能容纳

【同岗位对标】
  同岗级（P5 前端）在职员工 12 人，薪资区间 22k ~ 29k
  调整后将成为全公司 P5 前端最高薪（高出第二名 10%）

【倒挂风险】
  ⚠️ 该员工直属 Leader 为 P6，月薪 30k
  调整后员工薪资（32k）> Leader 薪资（30k），构成管理倒挂

【调整建议】（按优先级）
  方案 A（推荐）：同步晋升 P5→P6，薪资调至 30k（P6 下限）
    - 业绩优秀 + 晋升通道齐走，组织内逻辑自洽
    - 年度调薪 20%，仍在合理激励区间
  方案 B：保留 P5，薪资调至 29k（P5 带宽上限）
    - 待下半年晋升 P6 再做二次调整
  方案 C（当前方案）：保留 P5，直接给 32k
    - 风险：打破带宽设计、引发 Leader 倒挂、影响内部公平性
    - 仅在"特殊保留人才 + 高管特批"场景考虑

—— AI 诊断结果仅供参考，最终方案需薪酬委员会审定
```

#### 关键约束
- **非阻塞**：诊断卡不会阻止用户提交薪资，只是信息展示
- **颜色分级**：合理 🟢 / 偏离带宽 🟡 / 高风险 🔴
- **摘要 + 详情**：默认只展示一行摘要不打扰填表，需要深度看点"完整报告"

#### 市面对标
- Payscale MarketPay
- Workday Compensation
- SAP SuccessFactors Pay Equity
- Oracle HCM Compensation Workbench

## 四、统一技术基础

### 4.1 调用链路

```
业务页面按钮触发
      ↓
组件层（AIAssistPopover / AIAssistDialog / AISalarySanityInline）
      ↓
API 层 invokeAIAbility(code, input, operator, targetEmployee?)
      ↓
Mock 层 mockInvokeAI()  →  真实环境：调用 AI 凭证（GPT-4 / Claude / 文心一言）
      ↓
留痕：写入 aiCallRecordMock（调用记录表）
      ↓
返回输出 + recordId
      ↓
用户处置（采纳 / 修改采纳 / 放弃 / 草稿）
      ↓
updateAIAdoption(recordId, result, modificationDegree)
```

### 4.2 留痕表结构（AICallRecord）

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | number | 记录 ID |
| `abilityCode` | string | 能力编码 |
| `abilityName` | string | 能力名称 |
| `operatorName / operatorId` | string / number | 调用人 |
| `targetEmployee / targetEmployeeId` | string / number | 目标对象（如员工姓名）|
| `inputSummary` | string | 输入摘要（前 100 字）|
| `outputSummary` | string | 输出摘要（前 100 字）|
| `adoptionResult` | enum | `adopted` / `modified_adopted` / `rejected` / `draft` |
| `modificationDegree` | number | 修改程度（0~100%）|
| `callTime` | string | 调用时间 |
| `createTime` | string | 创建时间 |

### 4.3 采纳率追踪指标

AI 中心提供以下效果指标（管理员视角）：

| 指标 | 计算 | 业务含义 |
|---|---|---|
| **总调用量** | COUNT(调用记录)| AI 使用活跃度 |
| **采纳率** | `(adopted + modified_adopted) / total` | 整体效果（越高越好）|
| **原样采纳率** | `adopted / total` | AI 输出质量（越高说明 AI 越准）|
| **修改采纳率** | `modified_adopted / total` | AI 作为起点价值（说明 AI 省去了从 0 到 70% 的时间）|
| **放弃率** | `rejected / total` | AI 输出偏差（高放弃率说明需要调优）|
| **平均修改程度** | AVG(modificationDegree) | 员工对 AI 输出的信任度 |

### 4.4 AI 能力管理中心

**位置**：数据洞察 → AI 辅助中心（`/insight/ai-assistant`）

**功能**：
- **能力目录**：7 张卡片展示所有能力 + 成熟度 + 适用场景
- **业务入口跳转**：每张卡片点击"跳转到业务入口"直达真实触发点
- **演示调用**：管理员可演示能力（不影响真实业务数据）
- **调用记录审计**：按能力 / 操作人 / 时间维度查询留痕

### 4.5 AI 凭证管理

**位置**：招聘配置 → 外部集成

**管理内容**：
- LLM 提供商（GPT-4 / Claude / 文心一言 / 混元 / 通义）
- API Key（加密存储）
- 每月调用配额
- 各能力的模型路由（如 OKR 检查用 GPT-4，简历解析用 Claude）
