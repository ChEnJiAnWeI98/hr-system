## 一、重构目标与原则

### 1.1 重构目标

本次重构的**三个核心目标**（按优先级排序）：

1. **地基补齐**：建立"组织 / 岗位体系 / 员工档案 / RBAC 权限"四大地基，让所有业务模块有统一数据源和权限控制
2. **结构归位**：21 个一级菜单 → 8 个一级菜单，对标北森 / Workday / SAP SuccessFactors 等主流 HR SaaS
3. **六大模块完整化**：填补当前缺失的"人力规划 / 薪酬福利 / 培训与开发 / 员工关系（异动/离职）"等能力，达到对标头部产品的功能覆盖

### 1.2 四项原则

| 原则 | 具体含义 |
|---|---|
| **员工档案为中心** | 所有业务数据围绕员工档案辐射；员工档案是"唯一事实来源"（Single Source of Truth） |
| **状态机驱动** | 员工生命周期、合同、绩效周期、招聘流程、异动流程等全部用状态机建模 |
| **数据字典优先** | 先做字典（组织/岗位/状态/类型），再做业务页；业务页的选项全部从字典读取 |
| **权限分三层** | 菜单权限（能否看到入口）× 数据权限（能看哪些数据）× 字段权限（敏感字段是否显示） |

### 1.3 不做清单（明确砍掉）

- 候选人门户 / 招聘官网（面向外部候选人的公开页）
- 校招专项 / 猎头协作门户
- 员工自助独立一级菜单（改为角色视图切换）
- 在线学习播放器（视频/考试/章节进度等）
- 真实考勤打卡（硬件对接 / 钉钉飞书集成）
- 真实个税累进 / 五险一金地区规则计算
- 旧 URL 兼容 / Mock 数据兼容（全部重置）

## 二、六大模块 × 当前系统 × 新架构映射

### 2.1 传统六大 HR 模块

| 六大模块 | 核心能力 | 头部产品对标模块 |
|---|---|---|
| 1. 人力资源规划 | 组织架构演进 / 编制管理 / 人力预算 | Workday HCM / 北森 组织人事 |
| 2. 招聘与配置 | 需求 → 招聘 → 面试 → Offer → 入职 | SAP Recruiting + Onboarding |
| 3. 培训与开发 | 课程管理 / 培训计划 / 能力模型 / 继任 | SAP Learning + Succession |
| 4. 绩效管理 | 目标 → 评估 → 校准 → 档案 | SAP Performance & Goals |
| 5. 薪酬福利管理 | 薪资体系 / 核算 / 福利 / 社保 / 个税 | SAP Compensation |
| 6. 员工关系管理 | 合同 / 异动 / 离职 / 员工沟通 | Workday HCM - Employee Relations |

外加两个互联网时代的补充模块：
- **7. 数据洞察**（Workforce Analytics）：跨模块的分析与 AI 辅助
- **8. 系统管理**：RBAC / 审批流 / 字典 / 通知

共 **8 个一级菜单**。

### 2.2 六大模块 × 当前系统覆盖度

| 六大模块 | 当前状态 | 具体差距 |
|---|:---:|---|
| 人力资源规划 | ❌ 完全缺失 | 无"编制管理、HR 预算、组织规划" |
| 招聘与配置 | ✅ 较完整（90%） | 散落 4 个一级菜单；onboarding 重复 |
| 培训与开发 | ⚠️ 部分（30%） | 有 training 一级但较薄；能力模型错放绩效下；无继任 |
| 绩效管理 | ✅ 非常完整（100%） | 但散落 4 个一级菜单 |
| 薪酬福利管理 | ⚠️ 散装（25%） | salary + socialSecurity 散装；无薪资体系、无核算、无福利 |
| 员工关系管理 | ⚠️ 部分（35%） | contract 有（但位置错）；调动 / 离职 / 员工关系事件全缺 |
| 数据洞察 | ⚠️ 部分（50%） | dataReport 独立一级；各模块内还有分析，未统一 |
| 系统管理 | ⚠️ 部分（40%） | 有 system 但 RBAC 不完整；审批引擎独立一级 |

**总体覆盖度约 55%**。做完重构 + Phase 3/4 后可达 **95%+**。

### 01 组织人事（核心 HR） 🏢

**定位**：整个 HR 系统的"中央枢纽"，所有员工数据的唯一来源

**二级菜单（7 个）**：

| 二级 | 路径 | 关键功能 |
|---|---|---|
| 组织架构 | `/hrm/organization` | 组织树 CRUD / 组织属性（负责人/成本中心/编制） / 组织演进历史 / 组织迁移（含子组织） |
| 岗位体系 | `/hrm/position` | 岗位族 × 序列 × 职级矩阵 / 岗位说明书 / 任职资格 / 岗位等级映射 |
| 员工档案 | `/hrm/employee` | 花名册 / 员工详情（8 大 Tab）/ 批量导入 / 档案字段权限 |
| 合同管理 | `/hrm/contract` | 合同 CRUD / 续签提醒 / 电子签约 / 合同模板库 / 合同统计 |
| 人员异动 | `/hrm/transfer` | 调动 / 晋升 / 降级 / 调薪 / 异动审批流 / 异动报表 |
| 离职管理 | `/hrm/offboarding` | 离职申请 / 离职交接 / 离职面谈 / 离职证明 / 离职统计 |
| 编制管理 | `/hrm/headcount` | 编制预算 / HC 占用 / 编制调整 / 编制告警 |

**原 employee 改造要点**：
- 接入"统一员工状态机"（待入职→试用期→在职→调动中→离职中→已离职）
- 员工详情页改为 **8 大 Tab**：基本信息 / 岗位与组织 / 合同 / 薪酬 / 绩效 / 培训 / 异动历史 / 关联关系

## 四、地基层设计（核心）

这是重构的**灵魂部分**。所有后续模块的开发都依赖这里的数据模型。

### 4.1 组织架构数据模型

**组织节点类型**：
```typescript
type OrgNodeType =
  | 'group'      // 集团
  | 'company'    // 公司/子公司
  | 'division'   // 事业部
  | 'department' // 部门
  | 'team'       // 小组
  | 'virtual'    // 虚拟组织
```

**核心字段**：
- id / orgCode（组织编码，如 `G001-D005`）/ orgName / orgShortName
- parentId / level（层级）/ path（完整路径，便于权限判断）
- nodeType / orgNature（经营/职能/支持）
- managerId / managerName（组织负责人）
- hrbpId（对应 HRBP）
- costCenter（成本中心代码）
- headcount（编制数）/ actualCount（实际人数）
- startDate / endDate（生效时间）
- status（启用/停用/筹建中）
- sortOrder

**能力**：
- 组织树可视化（Tree）
- 组织迁移（带子组织和员工）
- 组织演进历史（成立/合并/拆分/撤销）

### 4.2 岗位体系（三层结构）

**层级关系**：
```
岗位族（Job Family）
  └ 岗位序列（Job Sequence）
      └ 岗位（Position / Job Title）
          └ 职级（Level）
```

**岗位族**（固定 6 类 + 可扩展）：
- 技术 / 产品 / 销售 / 运营 / 职能 / 管理

**岗位序列**（每个族下细分）：
- 技术族：后端 / 前端 / 算法 / 测试 / 运维 / 架构
- 产品族：产品经理 / 产品设计 / 交互 / UI
- ......

**岗位**（具体岗位名）：
- 如"高级后端工程师"

**职级体系**：
- 专业序列：P1~P10（可配置）
- 管理序列：M1~M5（可配置）
- 每个职级关联：薪酬带宽 / 任职资格 / 能力要求

**核心字段（岗位）**：
- id / positionCode / positionName
- jobFamily / jobSequence / levelSpec（适用职级范围，如 P5~P7）
- jobDescription（职责描述）
- qualification（任职资格）
- status / effectiveDate

### 4.3 员工档案 + 生命周期状态机

**状态机**（核心）：
```
    [已录取]
        ↓
    [待入职] ──── 未入职即离职 ───→ [候选人池]
        ↓
    [试用期] ──── 试用期不通过 ───→ [已离职]
        ↓
    [在职-正式]
        ↓ ↑
    [调动中] ←→ [借调中]
        ↓
    [离职中（办理手续）]
        ↓
    [已离职] ─── 返聘 ──→ [试用期]（开启新档案）
```

**状态枚举**：
```typescript
type EmployeeStatus =
  | 'pending_onboard'  // 待入职
  | 'probation'        // 试用期
  | 'regular'          // 正式在职
  | 'transferring'     // 调动中
  | 'seconded'         // 借调中
  | 'offboarding'      // 离职中
  | 'terminated'       // 已离职
```

**档案字段分组（7 类 + 关联关系）**：

| 分组 | 字段 |
|---|---|
| 基本信息 | 姓名/性别/出生日期/证件类型/证件号/民族/国籍/婚姻/政治面貌/学历/毕业学校/专业/联系方式/紧急联系人 |
| 岗位与组织 | 员工编号/组织/岗位族/岗位序列/岗位/职级/上级/HRBP/直线领导/工作地点/入职日期/司龄/转正日期 |
| 合同 | 当前合同类型/起止时间/续签次数/试用期/试用期备注 |
| 薪酬 | 月薪基数/岗位工资/绩效基数/社保基数/薪酬带宽位置 |
| 绩效历史 | 历次绩效等级/综合分/晋升记录 |
| 培训 | 累计培训时长/证书/能力评估分 |
| 异动历史 | 入职/转正/调动/晋升/降级/调薪 |
| 关联关系 | 上级员工/下属员工/HRBP/直线领导/同部门员工 |

**字段权限规则**：
- 普通员工：只能看自己，敏感字段（薪酬/绩效历史）仅本人可见
- 部门负责人：看本部门，但薪酬字段隐藏
- HRBP：看分管事业部，可看薪酬
- HR 管理员：看全员，全字段可见
- 超级管理员：全部

### 4.5 RBAC 权限模型（三层）

#### 第一层：菜单权限
- 定义：每个角色能看到哪些菜单
- 数据模型：`role_menu (role_id, menu_id)`
- 实现：登录后根据角色动态生成侧边栏

#### 第二层：数据权限
- 定义：每个角色在某业务页面能看到哪些数据（**最关键，当前完全缺失**）
- 规则类型：
  - `self` - 仅本人
  - `subordinate` - 直属下属（不含自己）
  - `self_and_subordinate` - 本人 + 直属下属
  - `department` - 本部门（含子部门）
  - `department_exclusive` - 本部门（不含子部门）
  - `custom_org` - 自定义组织范围（如 HRBP 分管的事业部）
  - `all` - 全员

#### 第三层：字段权限
- 定义：某字段对某角色是否可见
- 典型场景：薪酬字段对"部门负责人"隐藏、身份证对"非 HR"隐藏

#### 预置角色（7 类）

| 角色编码 | 名称 | 菜单权限 | 典型数据权限 | 字段权限 |
|---|---|---|---|---|
| `super_admin` | 超级管理员 | 全部 | all | 全部可见 |
| `hr_admin` | HR 管理员 | 除"系统管理"外全部 | all | 全部可见 |
| `hr_bp` | HRBP | 除"系统管理"外全部 | custom_org | 薪酬字段可见 |
| `dept_manager` | 部门负责人 | 核心业务菜单 | department | 薪酬字段隐藏 |
| `line_manager` | 直线经理 | 核心业务 | self_and_subordinate | 薪酬字段隐藏 |
| `employee` | 普通员工 | 员工视图（档案/绩效/培训/考勤/工资条） | self | 薪酬仅本人 |
| `interviewer` | 面试官（临时角色） | 招聘-面试 | custom_resume | 简历字段 |

**核心原则**：
1. **员工档案是宇宙中心**：所有业务数据绑定 `employeeId`
2. **组织是骨架**：员工归属组织，组织决定数据权限
3. **岗位是规格**：员工的岗位族×序列×职级决定薪酬带宽、能力要求、晋升路径
4. **异动是事件**：所有员工状态变化都通过异动记录，形成完整履历
5. **RBAC 是过滤器**：同一份数据通过 RBAC 对不同角色呈现不同视图
