## 二、项目概述

### 2.1 项目背景

**内容要求：**
- 描述行业现状和面临的挑战
- 列出3-5个具体问题点，每个问题点单独成段，使用编号列表
- 说明项目建设的必要性和紧迫性
- 字数要求：300-500字

**示例结构：**
```markdown
随着[行业]的快速发展，[业务场景]面临着巨大挑战。传统的[管理方式]主要依赖[传统方式]，存在以下问题：

1. **问题一标题**：具体问题描述，说明问题的影响和后果
2. **问题二标题**：具体问题描述，说明问题的影响和后果
3. **问题三标题**：具体问题描述，说明问题的影响和后果
4. **问题四标题**：具体问题描述，说明问题的影响和后果
5. **问题五标题**：具体问题描述，说明问题的影响和后果

为解决上述问题，建设[项目名称]，通过[技术手段]，实现[核心价值]，为[目标用户]提供[核心功能]，提升[管理目标]的智能化水平。
```

### 2.2 项目建设目标

**内容要求：**
- 列出3-5个具体、可衡量的目标
- 每个目标单独成段，使用编号列表
- 每个目标包含：目标名称（加粗）+ 详细说明
- 说明目标的价值和预期效果
- 字数要求：每个目标80-150字

**示例结构：**
```markdown
本项目旨在建设一套[系统类型]，实现以下目标：

1. **目标一名称**：详细说明目标的内容、实现方式和预期效果，包括具体的功能点和业务价值

2. **目标二名称**：详细说明目标的内容、实现方式和预期效果，包括具体的功能点和业务价值

3. **目标三名称**：详细说明目标的内容、实现方式和预期效果，包括具体的功能点和业务价值

4. **目标四名称**：详细说明目标的内容、实现方式和预期效果，包括具体的功能点和业务价值

5. **目标五名称**：详细说明目标的内容、实现方式和预期效果，包括具体的功能点和业务价值
```

### 2.3 项目建设范围

**内容要求：**
- 如果项目分阶段建设，使用四级标题分别说明各阶段内容
- 明确包含的系统端（PC端、移动端等）
- 列出功能模块清单
- 说明数据范围
- 明确不包含的功能（如有）
- 字数要求：200-400字

**示例结构（分阶段项目）：**
```markdown
#### 2.3.1 第一阶段建设内容

本项目第一阶段建设范围包括：

**系统端**：
- PC端管理系统（Web应用）
- 移动端H5（微信/钉钉）

**功能模块**：
- 模块一：功能说明
- 模块二：功能说明
- 模块三：功能说明

**数据范围**：
- 数据类型一：数据说明
- 数据类型二：数据说明

#### 2.3.2 第二阶段建设内容

第二阶段在第一阶段基础上扩展以下功能：

**功能模块**：
- 扩展模块一：功能说明
- 扩展模块二：功能说明
```

**示例结构（单阶段项目）：**
```markdown
本项目建设范围包括：

**系统端**：
- PC端管理后台（Web应用）
- 移动端H5（微信/钉钉）

**功能模块**：
- 模块一：功能说明
- 模块二：功能说明
- 模块三：功能说明

**数据范围**：
- 数据类型一：数据说明
- 数据类型二：数据说明

**不包含的功能**：
- 功能一：说明原因
- 功能二：说明原因
```

### 2.4 需求分析

**内容要求：**
- 使用表格描述，支持表格合并
- 包含功能模块、子功能、需求描述三列
- 使用 `^` 符号合并相同功能模块的单元格
- 覆盖所有核心功能模块

**示例表格：**
```markdown
| 功能模块 | 子功能 | 需求描述 |
|---------|-------|---------|
| 用户管理 | 用户列表 | 查看所有用户信息，支持搜索和筛选 |
| ^ | 新增用户 | 添加新用户，填写基本信息 |
| ^ | 编辑用户 | 修改用户信息 |
| ^ | 删除用户 | 删除用户（软删除） |
| 角色管理 | 角色列表 | 查看所有角色信息 |
| ^ | 权限配置 | 为角色分配功能权限 |
```

### 2.5 业务流程图

**内容要求：**
- **必须调用 diagram-generator 技能生成**
- 包括系统整体流程图、业务流程图（泳道）、数据交互流程等
- 如果涉及复杂计算逻辑，需要详细说明计算公式和规则
- 图片命名：`images/business-flow.png`

### 2.6 用户角色定义

**内容要求：**
- 定义各类用户及其权限
- 使用表格展示角色清单
- 包含角色名称、角色描述、主要职责、权限范围

**示例表格：**
```markdown
| 角色名称 | 角色描述 | 主要职责 | 权限范围 |
|---------|---------|---------|---------|
| 系统管理员 | 负责系统配置和用户管理 | 用户管理、权限配置、系统监控 | 全部功能 |
| 业务人员 | 使用系统处理日常业务 | 数据录入、查询、审批 | 业务功能 |
| 普通用户 | 查看和使用基础功能 | 数据查询、报表查看 | 只读权限 |
```

## 三、项目建设内容

### 3.1 总体功能架构

**内容要求：**
- **必须调用 diagram-generator 技能生成系统功能架构图**
- 展示完整的菜单结构和层级关系
- 图片命名：`images/system-architecture.png`
- ⚠️ **强制要求**：不得用文字列表或 Markdown 结构代替架构图

**菜单结构文字说明：**
- 只列出菜单中显示的功能，不列出隐藏页面（如详情页、编辑页）
- 菜单项必须标注层级：（一级菜单）、（二级菜单）等
- 不标注权限说明，权限信息统一在 3.2 页面访问权限表格中说明
- 禁止添加功能描述，只列出菜单名称和层级

**示例：**
```markdown
**菜单结构**

- 系统管理（一级菜单）
  - 用户管理（二级菜单）
  - 角色管理（二级菜单）
  - 部门管理（二级菜单）
- 业务管理（一级菜单）
  - 订单管理（二级菜单）
  - 客户管理（二级菜单）
```

### 3.2 页面访问权限

**内容要求：**
- 使用表格展示各角色对各功能模块的访问权限
- 权限类型：查看、新增、编辑、删除、导出等

**示例表格：**
```markdown
| 功能模块 | 系统管理员 | 业务人员 | 普通用户 |
|---------|-----------|---------|---------|
| 用户管理 | 全部权限 | 查看 | 无 |
| 角色管理 | 全部权限 | 无 | 无 |
| 订单管理 | 全部权限 | 全部权限 | 查看 |
| 客户管理 | 全部权限 | 查看、编辑 | 查看 |
```

### 3.3 功能详细设计

**内容要求：**
- 每个功能模块作为独立的四级标题（#### 3.3.1、#### 3.3.2...）
- 每个模块内部的子章节必须使用五级标题 `#####`

**标题层级规范：**

| Markdown | Word 样式 | 字号 | 用途示例 |
|---------|---------|------|---------|
| `##` | Heading 2 | 18pt | 三、项目建设内容 |
| `###` | Heading 3 | 16pt | 3.3 功能详细设计 |
| `####` | Heading 4 | 14pt | 3.3.1 系统登录 |
| `#####` | Heading 5 | 12pt | （一）功能需求概述 |

**每个功能模块的标准结构：**

```markdown
#### 3.3.1 [功能名称]

##### （一）功能需求概述

简要说明功能目的和价值

##### （二）用户操作流程

**⚠️ 强制要求：必须调用 diagram-generator 技能生成活动图**
- 图片命名：`images/[功能名]-activity.png`
- 活动图只描述用户行为和系统可见反馈，不写技术实现词汇
- 禁止：调用接口、API请求、数据库查询等技术词汇
- 正确：用户点击/输入/选择 + 系统提示/跳转/刷新

文字步骤说明（可选，作为补充）：
1. 用户执行操作A
2. 系统显示结果B
3. ...

##### （三）功能设计

**系统交互时序**

**⚠️ 强制要求：必须调用 diagram-generator 技能生成时序图**
- 图片命名：`images/[功能名]-sequence.png`
- 参与者固定4个：用户、[功能名]页面、后端服务、数据库
- 禁止使用英文别名（as Frontend、as Backend）

**核心功能**

列出所有功能点，使用列表或分段描述：
- 功能点1：说明
- 功能点2：说明
- 功能点3：说明

**数据字段**

使用表格展示业务字段定义：

| 字段名称 | 字段类型 | 长度限制 | 是否必填 | 说明 |
|---------|---------|---------|---------|------|
| 用户名 | 文本输入框 | 4-20位 | 是 | 用户登录账号 |
| 密码 | 密码输入框 | 6-20位 | 是 | 用户登录密码 |
| 角色 | 下拉选择 | - | 是 | 用户角色 |

**业务规则**

- 规则1：描述
- 规则2：描述
- 规则3：描述

##### （四）业务逻辑关联

说明与其他功能模块的数据依赖关系
```

**特殊情况处理：**
- **一级菜单下有多个二级功能**：使用子编号（如 3.3.5.1、3.3.5.2），每个子功能使用六级标题（######）
- **多端项目**：如果项目包含多个端（Web + App），功能详细设计应按端分组（3.3.1 Web端功能、3.3.2 App端功能）
- **单端项目**：如果只有一个端，直接写功能名称，不需要分端

## 五、非功能需求

### 5.1 性能需求

- 响应时间：页面加载时间 < 3秒
- 并发用户数：支持 XXX 并发用户
- 数据处理能力：每秒处理 XXX 条数据

### 5.2 安全需求

- 身份认证：支持多因素认证
- 数据安全：敏感数据加密存储
- 权限控制：基于角色的访问控制（RBAC）

### 5.3 可用性需求

- 系统可用性：99.9%
- 故障恢复：RTO < 1小时，RPO < 15分钟
- 数据备份：每日自动备份

### 5.4 兼容性需求

- 浏览器兼容性：Chrome、Firefox、Safari、Edge 最新版本
- 设备兼容性：PC、平板、手机
- 屏幕分辨率：1920x1080、1366x768、移动端自适应
