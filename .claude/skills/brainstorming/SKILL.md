# Brainstorming Ideas Into Designs

## Overview

通过自然对话将用户的想法或需求清单转化为完整的设计方案。支持两种启动方式：

1. **用户描述想法**：自由描述"我想做 xxx"
2. **用户提供需求清单**：提供简要的功能点列表

设计方案确认后保存文档，再询问是否生成详细需求说明书。

<HARD-GATE>
在展示设计方案并获得用户确认之前，禁止编写任何代码、生成文件或执行任何实现操作。
</HARD-GATE>

## 流程

```
启动 → 理解输入 → 澄清问题（逐个） → 提出方案 → 展示设计 → 用户确认
→ 生成保存任务清单 → 按任务清单分段保存文档 → 询问是否生成需求说明书 → （可选）调用 req-doc
```

## 第一步：理解输入

**两种输入模式，根据用户实际情况判断：**

**模式 A：用户描述想法**（"我想做 xxx"）

- 探索项目当前上下文（文件、文档等）
- 逐一提问澄清：目的、用户群体、关键功能、约束条件、成功标准
- 每次只问一个问题，优先给出选项供用户选择

**模式 B：用户提供需求清单**

- 先仔细审查清单，识别以下问题：
  - **遗漏**：是否缺少关键功能或场景
  - **模糊**：描述不够具体，无法直接实现
  - **矛盾**：不同条目之间存在冲突
  - **边界不清**：权限、异常处理、数据范围等未明确
- 针对发现的问题逐一提问澄清
- 补充缺失的需求维度（如用户角色、数据来源、性能要求等）
- 只有在需求清单完整且清晰后，才进入方案设计

## 第二步：提出方案

- 提出 2-3 种实现思路，附上优缺点对比
- 给出明确的推荐方案及理由
- 用对话式语言展示，不要过于技术化

## 第三步：展示设计方案

设计方案按模块分节展示，每节结束后询问用户是否有问题：

覆盖内容（按复杂度裁剪）：

- 功能范围与核心模块
- 界面/交互设计（如适用）
- 数据结构与流向
- 关键边界情况处理

展示完整设计后，统一询问：

> "以上是完整的设计方案，请问有什么需要调整的地方吗？"

## 第四步：确认并保存

用户确认没有问题后：

1. 告知用户："好的，将把设计方案保存下来。"
2. 生成保存任务清单，将文档内容拆分为多个小任务：

**任务清单格式：**

```markdown
## 📋 文档保存任务清单

共 X 个任务

| 序号 | 任务名称               | 状态      |
| ---- | ---------------------- | --------- |
| 1    | 创建文件并写入文档头部 | ⏳ 等待中 |
| 2    | 追加核心模块1内容      | ⏳ 等待中 |
| 3    | 追加核心模块2内容      | ⏳ 等待中 |
| 4    | 追加数据结构内容       | ⏳ 等待中 |
| 5    | 追加边界情况处理内容   | ⏳ 等待中 |
```

**状态标识：**

- ⏳ 等待中：任务未开始
- 🔄 保存中：正在保存该部分
- ✅ 已完成：保存成功

3. 按任务清单逐步保存文档到 `docs/YYYY-MM-DD-<系统名称>-v1.0.md`
   - 文件名格式：日期 + 系统名称（中文或英文）+ 版本号
   - 示例：`2026-03-28-危险厂区巡检系统-v1.0.md` 或 `2026-03-28-inspection-system-v1.0.md`
   - 版本号规则：
     - 首次创建使用 v1.0
     - 如果同名文件已存在，自动递增版本号（v1.1, v1.2...）
     - 重大修改时可升级主版本号（v2.0, v3.0...)
   - **⚠️ 保存策略（分段保存，避免错误）**：
     - 任务1：使用 Write 工具创建文件并写入文档头部（标题、项目概述、功能范围）
     - 任务2-N：使用 Edit 工具逐步追加各部分内容（读取文件末尾内容作为 old_string，拼接新内容作为 new_string）
     - 每次写入不超过 500 行，确保不会触发 "error writing file" 错误
     - **每完成一个任务，更新任务清单状态（⏳ → 🔄 → ✅）**
4. 保存完成后，询问：

> "设计文档已保存 ✓
>
> 是否需要继续使用 **req-doc** 生成详细的需求说明书？需求说明书会包含完整的功能描述、接口定义、数据模型等开发所需的详细规格。"

## 第五步：后续行动

- **用户选择生成需求说明书**：调用 `req-doc` 技能，以保存的设计文档为基础生成
- **用户选择跳过**：告知用户可随时通过 `/req-doc` 生成，结束本次流程

## 关键原则

- **每次只问一个问题**，避免信息轰炸
- **优先给选项**，比开放性问题更易回答
- **YAGNI 原则**：去掉不必要的功能，保持设计简洁
- **逐步确认**：分节展示设计，及时发现偏差
- **灵活调整**：用户有异议时随时回退修改
- **任务清单**：让用户清楚看到进度，避免等待焦虑
- **分段保存**：避免一次性写入过长内容导致错误

## ⚠️ 任务清单执行铁律（不可违反）

1. **必须先展示完整任务清单**，再开始任何写入操作
2. **每个任务开始前**：将状态更新为 🔄 保存中，并重新展示任务清单
3. **每个任务完成后**：将状态更新为 ✅ 已完成，并重新展示任务清单
4. **禁止跳过状态更新**：即使任务很简单，也必须更新状态后再继续下一个
5. **禁止批量执行**：每次只执行一个任务，完成并更新状态后再执行下一个
6. **禁止一次性写入整个文档**：必须按任务清单分批用 Write/Edit 写入
7. **必须先创建文件再追加**：任务1 必须用 Write 工具创建文件并写入骨架，后续任务才能用 Edit 追加内容，禁止在文件不存在时直接用 Edit
